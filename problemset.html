Top-K IPs from access logs

~30 min Parsing Warm-up
Prompt

Given a path to an Apache/nginx-style access log file, return the top K IP addresses by request count. Each line looks like:

192.168.1.1 - - [10/Oct/2023:13:55:36 +0000] "GET /api/v1/users HTTP/1.1" 200 1234

Some lines will be malformed. The file may be large enough that loading it all into memory is not ideal. Return results sorted by count descending; ties broken by IP lexicographically.

Clarify
  • What constitutes "malformed"? (Missing fields, bad IP format, both?)
  • Should IPv6 be supported?
  • Is K always small relative to unique IPs? (Affects whether we keep all counts or use a min-heap.)
  • Is the file gzipped? Streaming over network?
Hints
  1. Stream the file line-by-line — never load it all into memory.
  2. The first whitespace-separated token is the IP. Extract it without parsing the full log structure.
  3. Use a HashMap<String, Long> for counts, then a PriorityQueue of size K for top-K extraction. O(N log K).
  4. Skip malformed lines silently but track a count — interviewers love seeing observability instincts.
Approach
▸ Design

Three sub-problems, each with a textbook answer:

  • Reading the file — the file might be huge, so streaming line-by-line wins over slurping it whole. Memory stays constant in the file size; the bottleneck becomes disk I/O, which is fine.
  • Counting — a hash map from IP → count. O(1) amortized per line, total O(N) where N is the line count.
  • Selecting top-K — the naïve sort is O(U log U) where U is the number of unique IPs. A min-heap of size K does it in O(U log K), which matters when K ≪ U (the common case — top 10 from millions of unique addresses).

The min-heap trick: keep a heap of size K with the smallest count at the top. For each candidate, push it; if the heap exceeds K, pop the smallest. After processing all candidates, the heap holds the top K.

Tie-breaking is the subtle part. The problem says ties go to the lexicographically smaller IP. So in the heap's ordering, we want the smaller IP to survive when counts are equal — meaning the larger IP should bubble to the top (be evicted first). That's why the comparator inverts the IP order.

Malformed lines are silently skipped, but the count is logged. This is a small detail interviewers love: it shows you're thinking about observability without being asked.

Solution
import java.io.*;
import java.nio.file.*;
import java.util.*;
import java.util.stream.*;

public class TopKIps {

    public record IpCount(String ip, long count) {}

    public static List<IpCount> topK(Path logFile, int k) throws IOException {
        if (k <= 0) throw new IllegalArgumentException("k must be positive");

        Map<String, Long> counts = new HashMap<>();
        long malformed = 0;

        try (BufferedReader reader = Files.newBufferedReader(logFile)) {
            String line;
            while ((line = reader.readLine()) != null) {
                String ip = extractIp(line);
                if (ip == null) { malformed++; continue; }
                counts.merge(ip, 1L, Long::sum);
            }
        }

        if (malformed > 0) {
            System.err.printf("skipped %d malformed lines%n", malformed);
        }

        // Min-heap of size k. Smaller counts get evicted first;
        // for ties, the lexicographically LARGER IP gets evicted
        // (so the smaller IP stays — matching our final tiebreak rule).
        Comparator<IpCount> heapOrder = Comparator
            .<IpCount>comparingLong(IpCount::count)
            .thenComparing(IpCount::ip, Comparator.reverseOrder());

        PriorityQueue<IpCount> heap = new PriorityQueue<>(heapOrder);

        for (var entry : counts.entrySet()) {
            heap.offer(new IpCount(entry.getKey(), entry.getValue()));
            if (heap.size() > k) heap.poll();
        }

        List<IpCount> result = new ArrayList<>(heap);
        result.sort(Comparator
            .comparingLong(IpCount::count).reversed()
            .thenComparing(IpCount::ip));
        return result;
    }

    private static String extractIp(String line) {
        if (line == null || line.isBlank()) return null;
        int sp = line.indexOf(' ');
        if (sp <= 0) return null;
        String token = line.substring(0, sp);
        return isValidIp(token) ? token : null;
    }

    private static boolean isValidIp(String s) {
        String[] parts = s.split("\\.");
        if (parts.length != 4) return false;
        for (String p : parts) {
            try {
                int n = Integer.parseInt(p);
                if (n < 0 || n > 255) return false;
            } catch (NumberFormatException e) { return false; }
        }
        return true;
    }
}

#include <algorithm>
#include <cstdint>

struct IpCount { std::string ip; long long count; };

static bool isValidIp(const std::string& s) {
    int dots = 0, n = 0; bool hasDigit = false;
    for (char c : s) {
        if (c == '.') {
            if (!hasDigit || n > 255) return false;
            ++dots; n = 0; hasDigit = false;
        } else if (c >= '0' && c <= '9') {
            n = n * 10 + (c - '0');
            hasDigit = true;
            if (n > 255) return false;
        } else return false;
    }
    return dots == 3 && hasDigit;
}

std::vector<IpCount> topK(const std::string& path, int k) {
    if (k <= 0) throw std::invalid_argument("k must be positive");

    std::ifstream in(path);
    if (!in) throw std::runtime_error("open failed");

    std::unordered_map<std::string, long long> counts;
    long long malformed = 0;
    std::string line;
    while (std::getline(in, line)) {
        auto sp = line.find(' ');
        if (sp == std::string::npos || sp == 0) { ++malformed; continue; }
        std::string ip = line.substr(0, sp);
        if (!isValidIp(ip)) { ++malformed; continue; }
        ++counts[std::move(ip)];
    }
    if (malformed) std::cerr << "skipped " << malformed << " malformed lines\n";

    // Min-heap of size k. Smaller counts evicted first; for ties,
    // the lexicographically LARGER ip is evicted (smaller ip wins).
    auto heapCmp = [](const IpCount& a, const IpCount& b) {
        if (a.count != b.count) return a.count > b.count;
        return a.ip < b.ip;
    };
    std::priority_queue<IpCount, std::vector<IpCount>, decltype(heapCmp)> heap(heapCmp);

    for (auto& [ip, c] : counts) {
        heap.push({ip, c});
        if ((int)heap.size() > k) heap.pop();
    }

    std::vector<IpCount> result;
    result.reserve(heap.size());
    while (!heap.empty()) { result.push_back(heap.top()); heap.pop(); }
    std::sort(result.begin(), result.end(),
        [](const auto& a, const auto& b) {
            if (a.count != b.count) return a.count > b.count;
            return a.ip < b.ip;
        });
    return result;
}
▸ Discussion / Follow-ups (with answers)
  • How do you scale to 10× the data? On a single machine, the approach is unchanged — disk I/O dominates and the algorithm is already linear. For multi-machine: shard the file by hash, count per shard, merge counters across shards, then top-K on the merged map. If approximate is acceptable, Count-Min Sketch with a small fixed memory footprint gives bounded-error counts regardless of cardinality. Top-K with sketches is a well-studied problem; the Misra-Gries summary is the textbook algorithm.
  • How do you handle concurrent writers (live tail of a log)? Replace the hash map with a concurrent map (ConcurrentHashMap in Java, a sharded map or tbb::concurrent_hash_map in C++). Use the atomic update primitives (compute, merge) rather than read-modify-write to avoid races. For very high write rates, per-thread local maps with periodic merging (à la LongAdder) reduces contention significantly.
  • Why not Stream.sorted().limit(K) or std::partial_sort? Both are O(N log N) in the worst case for the full sort variant, or O(N log K) for partial-sort. The min-heap is also O(N log K), so they're equivalent on paper. The heap wins when you don't want to materialize the full count list — for streaming top-K it's natural.
  • What if the file is gzipped? Wrap the input stream: GZIPInputStream in Java, boost::iostreams::filtering_istream with a gzip filter in C++. The parsing logic is unchanged. Worth mentioning that decompression is single-threaded for a single .gz file — for parallel decompression you need split files or a parallel codec like pigz/zstd.
  • How would you handle IPv6? The validator's the only thing to change. The structural approach is the same — first whitespace-separated token, validate it's a parseable address. InetAddress.getByName (Java) or inet_pton (C++) handle both v4 and v6 if you want a more permissive validator.
  • What if you only need approximate top-K with bounded memory? The Misra-Gries summary tracks K candidate counters and converges to the true top-K with bounded error proportional to N/K. Implementations are short (~30 lines) and the technique is what most database engines use internally for "top frequent values" estimation.

Sliding-window error rate

~40 min Parsing Medium
Prompt

You're building a small monitoring tool. Given a stream of request events, each with a timestamp (epoch millis) and an HTTP status code, produce a function that returns the error rate (4xx + 5xx as a fraction of total) over the last N seconds. The function will be called frequently — make it fast.

Events arrive roughly in order but not strictly so. Old events (older than the window) should be evicted automatically.

Clarify
  • What's the expected event rate? (Affects whether per-event eviction is fine or we need bucketing.)
  • Is "error" only 5xx, or also 4xx? Is 429 an error?
  • How out-of-order can events be? (1s? 1 minute?)
  • Single-threaded calls or concurrent reads/writes?
Hints
  1. Naïve approach: Deque<Event>, push on record, pop expired on every query. Fine for low rates.
  2. Better: bucket by second. Maintain a ring of buckets each holding (total, errors). O(1) writes and queries when window is in seconds.
  3. Out-of-order events: bucket index = floor(timestamp / 1000) % windowSize, but you need to detect stale buckets and reset them when the second wraps.
  4. For thread-safety, the simple deque version with a single lock is often plenty.
Approach
▸ Design

The naïve approach — a deque of every event with timestamps, popping expired ones on each query — works but grows unboundedly with traffic. A high-QPS service can fill it faster than queries can drain it.

The fix is bucketing. Quantize timestamps to second-resolution buckets and keep a fixed-size ring of (total, errors) counters, one per second of the window. A 60-second window needs 60 buckets — total memory is bounded regardless of QPS.

The clever part is handling bucket reuse. When the second wraps around (e.g., second 137 lands on the same array slot as second 77 in a 60-second window), we detect the staleness via the stored epochSec and reset the bucket. This avoids needing a separate "clear" pass.

Out-of-order events within the window are handled correctly because each event finds its own bucket by timestamp. Events older than the window get written into stale buckets that the query then ignores (because their epochSec is outside the query range).

Complexity: O(1) writes, O(W) queries where W is the window size in buckets. For larger windows, queries can be made O(1) too by maintaining running sums and updating them on bucket reset.

Concurrency: a single mutex around the bucket array is the cleanest answer. Per-bucket locking is possible but rarely worth the complexity unless profiling shows the mutex is hot. LongAdder (Java) or sharded counters can help if it is.

Solution
import java.util.*;
import java.util.concurrent.locks.*;

// Bucketed sliding window. O(1) record, O(W) query (W = window seconds).
// For larger windows, query can also be O(1) by maintaining running sums.
public class ErrorRateWindow {

    private record Bucket(long epochSec, long total, long errors) {}

    private final int windowSec;
    private final Bucket[] buckets;
    private final ReentrantLock lock = new ReentrantLock();

    public ErrorRateWindow(int windowSec) {
        if (windowSec <= 0) throw new IllegalArgumentException();
        this.windowSec = windowSec;
        this.buckets = new Bucket[windowSec];
        Arrays.fill(this.buckets, new Bucket(-1, 0, 0));
    }

    public void record(long tsMillis, int status) {
        long sec = tsMillis / 1000;
        int idx = (int) Math.floorMod(sec, windowSec);
        boolean isError = status >= 400;

        lock.lock();
        try {
            Bucket b = buckets[idx];
            if (b.epochSec() != sec) {
                // stale bucket -- this second wrapped around, reset it
                buckets[idx] = new Bucket(sec, 1, isError ? 1 : 0);
            } else {
                buckets[idx] = new Bucket(sec, b.total() + 1,
                    b.errors() + (isError ? 1 : 0));
            }
        } finally { lock.unlock(); }
    }

    public double errorRate(long nowMillis) {
        long nowSec = nowMillis / 1000;
        long oldest = nowSec - windowSec + 1;
        long total = 0, errors = 0;

        lock.lock();
        try {
            for (Bucket b : buckets) {
                if (b.epochSec() >= oldest && b.epochSec() <= nowSec) {
                    total += b.total();
                    errors += b.errors();
                }
            }
        } finally { lock.unlock(); }

        return total == 0 ? 0.0 : (double) errors / total;
    }
}

#include <atomic>
#include <cstdint>
#include <mutex>
#include <stdexcept>
#include <vector>

class ErrorRateWindow {
    struct Bucket {
        int64_t epochSec = -1;
        int64_t total = 0;
        int64_t errors = 0;
    };

    int windowSec_;
    std::vector<Bucket> buckets_;
    std::mutex mu_;

public:
    explicit ErrorRateWindow(int windowSec)
        : windowSec_(windowSec), buckets_(windowSec) {
        if (windowSec <= 0) throw std::invalid_argument("windowSec");
    }

    void record(int64_t tsMillis, int status) {
        int64_t sec = tsMillis / 1000;
        // Positive modulo even for negative timestamps
        int idx = static_cast<int>(((sec % windowSec_) + windowSec_) % windowSec_);
        bool isError = status >= 400;

        std::lock_guard<std::mutex> lock(mu_);
        auto& b = buckets_[idx];
        if (b.epochSec != sec) {
            b = {sec, 1, isError ? 1 : 0};
        } else {
            b.total++;
            if (isError) b.errors++;
        }
    }

    double errorRate(int64_t nowMillis) {
        int64_t nowSec = nowMillis / 1000;
        int64_t oldest = nowSec - windowSec_ + 1;
        int64_t total = 0, errors = 0;

        std::lock_guard<std::mutex> lock(mu_);
        for (auto& b : buckets_) {
            if (b.epochSec >= oldest && b.epochSec <= nowSec) {
                total += b.total;
                errors += b.errors;
            }
        }
        return total == 0 ? 0.0 : static_cast<double>(errors) / total;
    }
};
▸ Discussion / Follow-ups (with answers)
  • How do you make queries O(1)? Maintain running totals (totalSum, totalErrors) updated on every record and on every bucket reset. When a stale bucket gets reused, subtract its old (total, errors) from the running totals before overwriting. Queries then just read the totals. Trade-off: more bookkeeping on writes, but the writes were already O(1).
  • How do you handle sub-second windows? Same structure with millisecond-resolution buckets. A 1-second window with 10ms buckets needs 100 slots. Memory scales linearly with resolution; pick based on how spiky your traffic is. For 100ms+ windows on a typical service, second-resolution is plenty.
  • What about lock contention at very high QPS? Three options, in order of effort: (1) LongAdder per bucket field (Java) or sharded counters (C++) — eliminates contention at the cost of a more expensive read. (2) StampedLock with optimistic reads — readers don't block writers. (3) Per-bucket atomics with explicit memory ordering. In practice, option (1) covers 99% of real loads.
  • How do you handle very out-of-order events? Up to one window of skew is handled correctly by design — a late event finds its own bucket. Events older than a full window get written into stale buckets that are then ignored by queries. If you need true late-arrival accounting (e.g., events arriving 10 minutes late in a 1-minute window), you need a different structure — typically a sliding-window log retention with explicit late-data side outputs (the streaming-systems "watermark" model).
  • Should 4xx really count as errors? Depends on what you're measuring. Service health: 5xx only. Client experience: 4xx + 5xx. 429 (rate-limited) is debatable — it indicates traffic-shaping working as intended, not service degradation. Real implementations often expose this as a configurable predicate (e.g., isError: status -> status >= 500 || status == 429) and let the caller decide.
  • How do you compute percentile latencies, not just rates? Different problem entirely — count buckets aren't enough. You need a sketch like HdrHistogram or t-digest per bucket. They give bounded-error percentile estimates with ~kilobytes of memory and merge across buckets correctly.
№ 03

Thread-safe LRU cache

~40 min Primitive Medium
Prompt

Implement a fixed-capacity LRU cache with get(key) and put(key, value). Both must be O(1). The cache must be safe to use from multiple threads. Optional: support a per-entry TTL (expired entries should not be returned by get).

Clarify
  • Strict O(1) under concurrency, or amortized? (Strict O(1) with concurrency is hard; amortized is realistic.)
  • Should get on an expired entry just return null, or also evict?
  • Are nulls valid values, or are they "not found"?
  • What's the read/write ratio? (Affects whether to use ReadWriteLock vs single mutex.)
Hints
  1. Classic structure: HashMap<K, Node> + doubly-linked list. Move-to-front on access.
  2. For thread-safety, the simplest correct answer is a single ReentrantLock wrapping all mutations. Don't reach for lock-free unless asked.
  3. For bonus points: mention LinkedHashMap with accessOrder=true as a quick-and-dirty alternative.
  4. TTL: store expiresAt on each node, check on get.
Approach
▸ Design

The standard LRU structure is hash map + doubly-linked list. The hash map gives O(1) key lookup; the list gives O(1) insertion and removal anywhere. Combined, get and put are both O(1).

The list is ordered MRU-to-LRU (most-recently-used at the head, least-recently-used at the tail). On get or put, the touched node moves to the head. When the cache exceeds capacity, the tail node is evicted.

Sentinel nodes at head and tail simplify the linked-list operations enormously. Without them, every insert/remove needs special-cases for empty list, single-element list, head insertion, tail removal. With them, every operation is just "splice this node between two others" — no branches.

Concurrency: a single mutex is the right answer for a mid-level interview. Read-write locks look tempting but don't help — every get still mutates the list (move-to-front), so it's a write. Striped locking (sharding by hash bucket) is the next step up; full lock-free is rarely justified outside libraries like Caffeine.

TTL piggybacks naturally: store an expiresAt on each node and check it on get. Expired entries get evicted lazily on access. A separate sweeper thread for proactive expiration is a follow-up — not needed for correctness.

Quick alternative worth mentioning: Java's LinkedHashMap with accessOrder=true and an overridden removeEldestEntry gives you LRU in five lines. Good for showing breadth, but doesn't support TTL or fine-grained concurrency.

Solution
import java.util.*;
import java.util.concurrent.locks.*;

public class LruCache<K, V> {

    private static final class Node<K, V> {
        final K key;
        V value;
        long expiresAt;          // Long.MAX_VALUE = no TTL
        Node<K, V> prev, next;
        Node(K k, V v, long exp) { key = k; value = v; expiresAt = exp; }
    }

    private final int capacity;
    private final Map<K, Node<K, V>> map = new HashMap<>();
    private final Node<K, V> head = new Node<>(null, null, 0);  // MRU sentinel
    private final Node<K, V> tail = new Node<>(null, null, 0);  // LRU sentinel
    private final ReentrantLock lock = new ReentrantLock();

    public LruCache(int capacity) {
        if (capacity <= 0) throw new IllegalArgumentException();
        this.capacity = capacity;
        head.next = tail; tail.prev = head;
    }

    public V get(K key) {
        lock.lock();
        try {
            Node<K, V> n = map.get(key);
            if (n == null) return null;
            if (System.currentTimeMillis() >= n.expiresAt) {
                evict(n);
                return null;
            }
            moveToFront(n);
            return n.value;
        } finally { lock.unlock(); }
    }

    public void put(K key, V value) { put(key, value, Long.MAX_VALUE); }

    public void put(K key, V value, long ttlMillis) {
        long exp = ttlMillis == Long.MAX_VALUE
            ? Long.MAX_VALUE
            : System.currentTimeMillis() + ttlMillis;

        lock.lock();
        try {
            Node<K, V> existing = map.get(key);
            if (existing != null) {
                existing.value = value;
                existing.expiresAt = exp;
                moveToFront(existing);
                return;
            }
            Node<K, V> n = new Node<>(key, value, exp);
            map.put(key, n);
            addToFront(n);
            if (map.size() > capacity) evict(tail.prev);
        } finally { lock.unlock(); }
    }

    private void addToFront(Node<K, V> n) {
        n.next = head.next; n.prev = head;
        head.next.prev = n; head.next = n;
    }
    private void unlink(Node<K, V> n) {
        n.prev.next = n.next; n.next.prev = n.prev;
    }
    private void moveToFront(Node<K, V> n) { unlink(n); addToFront(n); }
    private void evict(Node<K, V> n) { unlink(n); map.remove(n.key); }
}
// C++17. Thread-safe LRU cache with optional TTL.
#include <chrono>
#include <list>
#include <mutex>
#include <optional>
#include <unordered_map>

template <class K, class V>
class LruCache {
    struct Entry {
        K key;
        V value;
        std::chrono::steady_clock::time_point expiresAt;
    };
    using It = typename std::list<Entry>::iterator;

    size_t capacity_;
    std::list<Entry> lru_;            // front = MRU, back = LRU
    std::unordered_map<K, It> map_;
    std::mutex mu_;

public:
    explicit LruCache(size_t capacity) : capacity_(capacity) {
        if (capacity == 0) throw std::invalid_argument("capacity");
    }

    std::optional<V> get(const K& key) {
        std::lock_guard<std::mutex> lock(mu_);
        auto it = map_.find(key);
        if (it == map_.end()) return std::nullopt;
        if (std::chrono::steady_clock::now() >= it->second->expiresAt) {
            lru_.erase(it->second);
            map_.erase(it);
            return std::nullopt;
        }
        lru_.splice(lru_.begin(), lru_, it->second);
        return it->second->value;
    }

    void put(K key, V value,
             std::chrono::milliseconds ttl = std::chrono::milliseconds::max()) {
        auto expiresAt = (ttl == std::chrono::milliseconds::max())
            ? std::chrono::steady_clock::time_point::max()
            : std::chrono::steady_clock::now() + ttl;

        std::lock_guard<std::mutex> lock(mu_);
        auto it = map_.find(key);
        if (it != map_.end()) {
            it->second->value = std::move(value);
            it->second->expiresAt = expiresAt;
            lru_.splice(lru_.begin(), lru_, it->second);
            return;
        }
        lru_.push_front({key, std::move(value), expiresAt});
        map_[std::move(key)] = lru_.begin();
        if (map_.size() > capacity_) {
            map_.erase(lru_.back().key);
            lru_.pop_back();
        }
    }
};
▸ Discussion / Follow-ups (with answers)
  • How do you reduce contention on read-heavy workloads? Counterintuitively, a read-write lock doesn't help — every get mutates the linked list (move-to-front), so all "reads" are actually writes. Real solutions: (a) Striped locking — shard by key hash into N independent caches each with its own lock. Reduces contention by N. (b) Caffeine's approach (Java) — buffer access events in per-thread ring buffers, replay them in batches under the cache lock. Decouples the hot path from the eviction state machine. (c) Approximate LRU using CLOCK or 2Q algorithms with atomic flags instead of a linked list.
  • What about LinkedHashMap with accessOrder=true? Five lines of code, gives you basic LRU. Limitations: not thread-safe (wrap with Collections.synchronizedMap), no TTL, can't easily customize eviction. Fine for small in-process caches; not for hot-path service caches. C++ has no direct equivalent — you'd build it.
  • What's the production-grade answer? Java: Caffeine. Window-TinyLFU eviction (smarter than LRU on real workloads), async refresh, statistics, weight-based sizing, automatic cleanup. C++: folly::EvictingCacheMap, boost::compute::detail::lru_cache, or roll your own — ecosystem is more fragmented. Always mention the library in interviews; it shows you know not to reinvent.
  • How do you handle memory pressure? Two extensions. (a) Weight-based sizing — entries declare their size in bytes, eviction is by total weight rather than count. Caffeine supports this directly. (b) Soft references — entries are GC-eligible under memory pressure (Java only; C++ has no equivalent). Soft refs give worst-case-throughput rather than worst-case-memory; they're a tail-latency footgun in production.
  • What if you need TTL with proactive expiration? Lazy expiration (the design here) is correct but lets dead entries accumulate. Active expiration runs a periodic sweeper or maintains a separate min-heap by expiry time. Caffeine uses a hierarchical timing wheel — O(1) insertion and expiration with bounded slot count. Implementing it is overkill for an interview but worth knowing the term.
  • What's the difference between LRU and LFU? LRU evicts the least-recently-used; LFU evicts the least-frequently-used. LFU resists scan-based pollution (a one-time bulk read doesn't evict your hot data) but needs frequency tracking. TinyLFU combines both: a sketch tracks frequency, the cache uses LRU as the base structure, and admission decisions consult the sketch. This is what Caffeine does and why it beats pure LRU on most real workloads.

Token-bucket rate limiter

~35 min Primitive Medium
Prompt

Build a per-user rate limiter using the token bucket algorithm. API: boolean tryAcquire(String userId). Each user has a bucket of capacity C tokens that refills at R tokens per second. tryAcquire returns true if a token was available and consumed, false otherwise. Thread-safe.

Clarify
  • Same limit for every user, or per-tier limits?
  • What happens to inactive users — is unbounded memory growth a concern?
  • Single-instance, or distributed across servers? (Distributed needs Redis-backed.)
  • Should we charge variable token costs (e.g. expensive endpoints cost more)?
Hints
  1. Don't use a background thread to refill. Compute tokens lazily on each request from elapsed time since last update.
  2. Per-user bucket state in a ConcurrentHashMap. Use computeIfAbsent + per-bucket synchronization.
  3. For inactive-user cleanup, mention TTL eviction or a Caffeine cache as the bucket store.
Approach
▸ Design

Token bucket has two parameters: capacity C (max burst) and refill rate R (tokens per second). The bucket starts full; every request consumes a token; tokens replenish over time up to C.

The naïve implementation uses a background thread that refills every bucket every tick. Don't do that. With millions of users, the refiller becomes a CPU sink and a contention point. Instead, refill lazily: on each tryAcquire, compute how much time has elapsed since the last update and add the corresponding tokens (capped at capacity). Zero background work, zero idle cost.

Per-user buckets live in a concurrent map keyed by user ID. The map provides safe concurrent insertion. Each bucket has its own lock — synchronizing on the bucket object itself scopes contention to a single user, so user A's traffic doesn't block user B's.

Why nanoTime / steady_clock instead of wall-clock time? Both are monotonic — they never go backward, even if the system clock gets adjusted by NTP. Time-since calculations using wall-clock can produce negative deltas during clock corrections, which corrupts the token math.

Why double for tokens? Allows fractional refill rates (e.g., 0.5 req/sec for slow APIs) and variable-cost permits (an "expensive" endpoint can charge 5 tokens per call).

Memory growth is the one issue this design has — inactive users' buckets stay forever. The fix is wrapping the map in a TTL cache (Caffeine, in Java; a simple LRU is fine in C++) so buckets for users idle longer than, say, 1 hour get evicted.

Solution
import java.util.concurrent.*;

public class TokenBucketLimiter {

    private static final class Bucket {
        double tokens;
        long lastRefillNanos;
        Bucket(double tokens, long nanos) {
            this.tokens = tokens;
            this.lastRefillNanos = nanos;
        }
    }

    private final double capacity;
    private final double refillPerNano;
    private final ConcurrentMap<String, Bucket> buckets = new ConcurrentHashMap<>();

    public TokenBucketLimiter(double capacity, double refillPerSecond) {
        if (capacity <= 0 || refillPerSecond <= 0) {
            throw new IllegalArgumentException();
        }
        this.capacity = capacity;
        this.refillPerNano = refillPerSecond / 1_000_000_000.0;
    }

    public boolean tryAcquire(String userId) { return tryAcquire(userId, 1.0); }

    public boolean tryAcquire(String userId, double permits) {
        long now = System.nanoTime();
        Bucket b = buckets.computeIfAbsent(userId,
            k -> new Bucket(capacity, now));

        // Per-bucket lock. Synchronizing on the bucket object itself keeps
        // contention scoped to a single user, not the whole limiter.
        synchronized (b) {
            long elapsed = now - b.lastRefillNanos;
            if (elapsed > 0) {
                b.tokens = Math.min(capacity, b.tokens + elapsed * refillPerNano);
                b.lastRefillNanos = now;
            }
            if (b.tokens >= permits) {
                b.tokens -= permits;
                return true;
            }
            return false;
        }
    }
}

    };

    double capacity_;
    double refillPerNano_;
    std::shared_mutex mapMu_;
    std::unordered_map<std::string, std::shared_ptr<Bucket>> buckets_;

    static int64_t nowNanos() {
        return std::chrono::duration_cast<std::chrono::nanoseconds>(
            std::chrono::steady_clock::now().time_since_epoch()).count();
    }

public:

        : capacity_(capacity),

        if (capacity <= 0 || refillPerSecond <= 0)
            throw std::invalid_argument("args");
    }

    bool tryAcquire(const std::string& userId, double permits = 1.0) {
        std::shared_ptr<Bucket> b;
        {
            std::shared_lock<std::shared_mutex> rd(mapMu_);
            auto it = buckets_.find(userId);
            if (it != buckets_.end()) b = it->second;

            b = slot;

        int64_t now = nowNanos();

        if (elapsed > 0) {
            b->tokens = std::min(capacity_, b->tokens + elapsed * refillPerNano_);
            b->lastRefillNanos = now;

        return false;

Retry with exponential backoff

~30 min Primitive Warm-up
Prompt

Implement a generic retry helper: <T> T executeWithRetry(Callable<T> op, RetryPolicy policy). It should retry transient failures with exponential backoff and full jitter. Non-retriable exceptions should propagate immediately. The policy specifies max attempts, base delay, max delay, and which exceptions are retriable.

Clarify
  • What constitutes "retriable"? (Exception type list, predicate, or explicit error codes?)
  • Full jitter, equal jitter, decorrelated jitter? (Default to full jitter — AWS recommendation.)
  • Should it support async (CompletableFuture) or just sync?
  • What about deadline-based stopping (total budget)?
Hints
  1. Full jitter: delay = random(0, min(max, base × 2^attempt)). Prevents thundering herd.
  2. Use InterruptedException handling correctly: re-set the interrupt flag.
  3. Builder pattern for RetryPolicy reads better than five-parameter constructors.
  4. Throw a wrapping exception with the last cause if all attempts exhausted — preserves the stack trace.
Approach
▸ Design

Three pieces:

  1. The retry loop: try, catch retriable, sleep, repeat up to N times.
  2. The backoff: exponential — delay roughly doubles each attempt — capped at a maximum.
  3. The jitter: randomize within the backoff window to prevent thundering herds.

Why jitter matters: if 1000 clients all retry exactly 100ms after a failure, they hit the recovering server simultaneously. With full jitter — sleep a uniform random time in [0, exp_backoff] — the retries spread out evenly across the window. AWS published a much-cited blog showing full jitter is empirically better than equal jitter or decorrelated jitter for most workloads.

What's "retriable"? Network timeouts, 5xx responses, connection failures — yes. Authentication errors, 4xx client errors, parse errors — no. The policy takes a predicate so the caller decides. A common bug is retrying everything, which masks real problems and amplifies bad client behavior.

Bit-shift overflow: 1L << attempt overflows when attempt ≥ 63. In practice, the min(maxDelay, ...) protects you, but explicit guarding (cap the shift at, say, 30) is cleaner. The C++ version caps at 62 to be safe.

Interruption: in Java, Thread.sleep can be interrupted. The correct response is to re-set the interrupt flag and propagate, not swallow the exception. C++ doesn't have the same model — sleep_for isn't interruptible by default — but condition_variable::wait_for is, if you need cancellable retries.

Stack-trace preservation: when retries are exhausted, wrap the last exception as the cause rather than swallowing it. Otherwise the operator sees "RetryExhausted" with no clue what was actually failing.

Solution
import java.time.*;
import java.util.concurrent.*;
import java.util.function.*;

public class Retry {

    public record Policy(
        int maxAttempts,
        Duration baseDelay,
        Duration maxDelay,
        Predicate<Throwable> isRetriable
    ) {
        public static Policy defaults() {
            return new Policy(3,
                Duration.ofMillis(100),
                Duration.ofSeconds(10),
                t -> t instanceof IOException ||
                     t instanceof TimeoutException);
        }
    }

    public static class RetryExhaustedException extends RuntimeException {
        public RetryExhaustedException(String msg, Throwable cause) {
            super(msg, cause);
        }
    }

    public static <T> T execute(Callable<T> op, Policy policy) throws Exception {
        Throwable last = null;
        for (int attempt = 0; attempt < policy.maxAttempts(); attempt++) {
            try {
                return op.call();
            } catch (InterruptedException ie) {
                Thread.currentThread().interrupt();
                throw ie;
            } catch (Exception e) {
                if (!policy.isRetriable().test(e)) throw e;
                last = e;
                if (attempt == policy.maxAttempts() - 1) break;
                sleepWithJitter(attempt, policy);
            }
        }
        throw new RetryExhaustedException(
            "giving up after " + policy.maxAttempts() + " attempts", last);
    }

    private static void sleepWithJitter(int attempt, Policy p)
            throws InterruptedException {
        long exp = Math.min(
            p.maxDelay().toMillis(),
            p.baseDelay().toMillis() * (1L << attempt));
        // Full jitter: uniform in [0, exp]
        long sleep = ThreadLocalRandom.current().nextLong(exp + 1);
        Thread.sleep(sleep);
    }
}

#include <chrono>
#include <exception>
#include <functional>
#include <random>
#include <stdexcept>
#include <thread>

struct RetryPolicy {
    int maxAttempts;
    std::chrono::milliseconds baseDelay;
    std::chrono::milliseconds maxDelay;
    std::function<bool(const std::exception&)> isRetriable;

    static RetryPolicy defaults() {
        return {3,
                std::chrono::milliseconds(100),
                std::chrono::seconds(10),
                [](const std::exception&) { return true; }};
    }
};

class RetryExhausted : public std::runtime_error {
public:
    std::exception_ptr last;
    RetryExhausted(const std::string& m, std::exception_ptr e)
        : std::runtime_error(m), last(e) {}
};

template <class F>
auto executeWithRetry(F&& op, const RetryPolicy& policy)
        -> decltype(op()) {
    std::exception_ptr last;
    std::mt19937_64 rng(std::random_device{}());

    for (int attempt = 0; attempt < policy.maxAttempts; ++attempt) {
        try {
            return op();
        } catch (const std::exception& e) {
            if (!policy.isRetriable(e)) throw;
            last = std::current_exception();
            if (attempt == policy.maxAttempts - 1) break;

            // Full jitter: uniform in [0, min(maxDelay, base * 2^attempt)]
            int64_t shift = std::min(62, attempt);   // guard overflow
            int64_t exp = std::min(policy.maxDelay.count(),
                                  policy.baseDelay.count() * (1LL << shift));
            std::uniform_int_distribution<int64_t> dist(0, exp);
            std::this_thread::sleep_for(std::chrono::milliseconds(dist(rng)));
        }
    }
    throw RetryExhausted("giving up", last);
}
▸ Discussion / Follow-ups (with answers)
  • Why full jitter beats equal jitter beats no jitter? AWS's classic blog post measured this empirically. No jitter: 1000 clients retry at exactly t+100ms — thundering herd hits the recovering server, often re-failing. Equal jitter (sleep base/2 + random(0, base/2)): herd is partially synchronized — improvement, but a noticeable spike remains. Full jitter (sleep random(0, base)): retries spread uniformly across the window, no spike. Counter-intuition: full jitter has higher variance per client, but lower correlation across clients, which is what matters for the recovering service.
  • How do you implement async retries? Sync retries block a thread during sleep — wasteful at scale. Async version: return CompletableFuture<T> (Java) or std::future<T> / folly::Future<T> (C++), schedule retry attempts on a ScheduledExecutorService or equivalent timer. The thread isn't blocked between attempts. This matters when you have thousands of in-flight retried operations — a thread per retry is a non-starter.
  • What about a total deadline (budget) instead of just attempt count? Add Duration totalBudget to the policy. Track elapsed time across attempts; abort if elapsed + nextDelay > budget. This is more useful than attempt count for user-facing operations — "retry for up to 5 seconds" maps directly to a UX requirement, "retry 3 times" doesn't.
  • What if the retried operation isn't idempotent? Don't retry it. Or wrap it in an idempotency key (the operation includes a client-generated UUID; the server dedupes by key). Retrying a non-idempotent operation can cause duplicate side effects — a charged credit card billed twice, an email sent twice. The retry policy should refuse to retry mutations unless the caller has guaranteed idempotency.
  • What's the production library answer? Java: Resilience4j (modular, modern, integrates with reactive types) or Failsafe (simpler API, fewer features). C++: rolling your own is more common, though Microsoft's PPL has retry-like primitives and folly::futures can compose retries via onError. Mention the library; show you wouldn't ship hand-rolled retry logic for production HTTP calls.
  • How do you handle retriable vs non-retriable in HTTP? Standard rules: 5xx is retriable (server problem, might recover). 408 (Request Timeout), 429 (Too Many Requests) are retriable with respect to the Retry-After header if present. 4xx other than those is not retriable — the request is bad and won't get better with retries. Connection errors (refused, reset, timeout) are retriable. The predicate in the policy encodes this; bake it as a default for HTTP clients.

CSV dedup & merge

~40 min Pipeline Medium
Prompt

You're given two CSV files of customer records with columns id, email, name, updated_at. Merge them into one deduplicated CSV. A duplicate is any record sharing the same id OR the same normalized email (lowercase, trimmed). When duplicates exist, keep the record with the latest updated_at. Drop rows with malformed timestamps or empty IDs.

Clarify
  • What's the timestamp format? (ISO-8601 assumed.)
  • How are conflicts resolved when ID matches one record but email matches another?
  • Are emails case-insensitive across the board, or does the casing on output matter?
  • Can the files have different headers, or are they guaranteed identical?
Hints
  1. Two-pass union-find when matching by both id and email gets messy. Simpler: keep two indexes, byId and byEmail, both pointing to the same record object.
  2. For each incoming record, look up both indexes. Resolve "winner" by updated_at, then unify both indexes onto the winner.
  3. Use OpenCSV or Apache Commons CSV for parsing — don't hand-roll, comma-in-quoted-field handling is a trap.
  4. Stream both files; don't load everything if data is huge.
Approach
▸ Design

The trap in this problem is the dual-key dedup. You're matching by id OR email, which means a single record can collide with multiple existing records — one by id, a different one by email — and the merge has to handle three-way reconciliation.

The clean answer: keep two indexes (byId and byEmail) that both point to the same record objects. When a new record arrives, look it up in both. There are up to three records in play (incoming + id-match + email-match), all potentially different. Pick the one with the latest updated_at as the winner, then atomically:

  • Remove the loser entries from whichever index they were in (otherwise stale pointers).
  • Install the winner in both indexes.

The output deduplication step uses identity-based set semantics (Java's IdentityHashMap, C++'s unordered_set<Customer*>) because two records can be .equals-equal as values but represent the same logical entity we want to write once.

Why ISO-8601 strings work directly for ts comparison in C++: lexicographic order matches chronological order for ISO-8601's fixed-width format. Saves parsing for the comparison hot path.

Don't hand-roll CSV parsing in production. Quoted commas, embedded newlines, escaped quotes, BOMs — they fail subtly and silently. The example code uses split for brevity; mention OpenCSV (Java) or a library like Vince's CSV parser (C++) in the conversation.

Streaming: if the files don't fit in memory, the structure changes. External merge sort by key (id, then email) lets you do a linear merge pass. Or push everything into a temp SQLite database — surprisingly often the right answer for "data wrangling at scale on one machine."

Solution
import java.io.*;
import java.nio.file.*;
import java.time.*;
import java.time.format.*;
import java.util.*;

public class CsvMerger {

    public record Customer(String id, String email,
                            String name, Instant updatedAt) {
        String normalizedEmail() { return email.trim().toLowerCase(Locale.ROOT); }
    }

    public static void merge(Path a, Path b, Path out) throws IOException {
        Map<String, Customer> byId = new HashMap<>();
        Map<String, Customer> byEmail = new HashMap<>();
        int rejected = 0;

        for (Path p : List.of(a, b)) {
            try (BufferedReader r = Files.newBufferedReader(p)) {
                String header = r.readLine();          // id,email,name,updated_at
                String line;
                while ((line = r.readLine()) != null) {
                    Customer c = parseRow(line);
                    if (c == null) { rejected++; continue; }
                    upsert(c, byId, byEmail);
                }
            }
        }

        // byId and byEmail may share entries; collect unique records by identity
        Set<Customer> unique = Collections.newSetFromMap(new IdentityHashMap<>());
        unique.addAll(byId.values());
        unique.addAll(byEmail.values());

        try (BufferedWriter w = Files.newBufferedWriter(out)) {
            w.write("id,email,name,updated_at\n");
            for (Customer c : unique) {
                w.write(String.format("%s,%s,%s,%s%n",
                    csvEscape(c.id()), csvEscape(c.email()),
                    csvEscape(c.name()), c.updatedAt()));
            }
        }
        System.err.printf("merged: %d rows, rejected: %d%n", unique.size(), rejected);
    }

    private static void upsert(Customer incoming,
                               Map<String, Customer> byId,
                               Map<String, Customer> byEmail) {
        Customer matchById = byId.get(incoming.id());
        Customer matchByEmail = byEmail.get(incoming.normalizedEmail());

        // Determine the winner among incoming + any matches
        Customer winner = incoming;
        if (matchById != null && matchById.updatedAt().isAfter(winner.updatedAt())) {
            winner = matchById;
        }
        if (matchByEmail != null && matchByEmail.updatedAt().isAfter(winner.updatedAt())) {
            winner = matchByEmail;
        }

        // Remove losers from both indexes, then install the winner
        if (matchById != null && matchById != winner) {
            byEmail.remove(matchById.normalizedEmail(), matchById);
        }
        if (matchByEmail != null && matchByEmail != winner) {
            byId.remove(matchByEmail.id(), matchByEmail);
        }
        byId.put(winner.id(), winner);
        byEmail.put(winner.normalizedEmail(), winner);
    }

    private static Customer parseRow(String line) {
        // Production: use a real CSV parser. This is a simplification.
        String[] f = line.split(",", -1);
        if (f.length != 4 || f[0].isBlank() || f[1].isBlank()) return null;
        try {
            Instant ts = Instant.parse(f[3].trim());
            return new Customer(f[0].trim(), f[1].trim(), f[2].trim(), ts);
        } catch (DateTimeParseException e) { return null; }
    }

    private static String csvEscape(String s) {
        return (s.contains(",") || s.contains("\""))
            ? "\"" + s.replace("\"", "\"\"") + "\""
            : s;
    }
}

struct Customer {
    std::string id, email, name, ts;     // ts as ISO-8601 string
    std::string normalizedEmail() const {
        std::string e = email;
        std::transform(e.begin(), e.end(), e.begin(),
                       [](unsigned char c) { return std::tolower(c); });
        // trim
        auto a = e.find_first_not_of(" \t");

        return a == std::string::npos ? "" : e.substr(a, b - a + 1);

// ISO-8601 lex order is chronological order; string > works.
static std::shared_ptr<Customer> parseRow(const std::string& line) {
    std::vector<std::string> f;
    std::stringstream ss(line);
    std::string tok;
    while (std::getline(ss, tok, ',')) f.push_back(tok);
    if (f.size() != 4 || f[0].empty() || f[1].empty() || f[3].empty())
        return nullptr;
    return std::make_shared<Customer>(Customer{f[0], f[1], f[2], f[3]});
}

void mergeCsvs(const std::string& a, const std::string& b,
               const std::string& outPath) {
    using CPtr = std::shared_ptr<Customer>;
    std::unordered_map<std::string, CPtr> byId, byEmail;
    int64_t rejected = 0;

    for (const auto& path : {a, b}) {
        std::ifstream in(path);
        std::string line;
        std::getline(in, line);                 // skip header
        while (std::getline(in, line)) {
            auto c = parseRow(line);
            if (!c) { ++rejected; continue; }

            auto mIt = byId.find(c->id);
            auto eIt = byEmail.find(c->normalizedEmail());
            CPtr winner = c;
            if (mIt != byId.end() && mIt->second->ts > winner->ts) winner = mIt->second;
            if (eIt != byEmail.end() && eIt->second->ts > winner->ts) winner = eIt->second;

            // Evict losers from the opposite index, then install winner
            if (mIt != byId.end() && mIt->second != winner)
                byEmail.erase(mIt->second->normalizedEmail());
            if (eIt != byEmail.end() && eIt->second != winner)
                byId.erase(eIt->second->id);
            byId[winner->id] = winner;
            byEmail[winner->normalizedEmail()] = winner;
        }
    }

    // Unique by pointer identity
    std::unordered_set<Customer*> seen;
    std::ofstream out(outPath);
    out << "id,email,name,updated_at\n";
    auto dump = [&](const auto& m) {
        for (auto& [_, c] : m)
            if (seen.insert(c.get()).second)
                out << c->id << ',' << c->email << ','
                    << c->name << ',' << c->ts << '\n';
    };
    dump(byId); dump(byEmail);
    std::cerr << "merged: " << seen.size() << ", rejected: " << rejected << '\n';

  • How do you parallelize? Two-stage. (1) Each worker reads its chunk and produces a partial dedup result (two indexes locally). (2) Merge step combines workers' indexes pairwise. The merge step has the same logic as the inner loop. The trick: chunks must be split on line boundaries — find a newline near the desired byte offset rather than splitting at exact byte counts.
  • What if email normalization needs to be more aggressive? Real-world: Gmail treats foo+bar@gmail.com and f.o.o@gmail.com as the same address. If the dedup needs to match these, normalization needs domain-specific rules (strip dots and plus-tags for @gmail.com; not for other domains). Streaming JSON aggregator

~45 min Pipeline Medium
Prompt

You're given a multi-gigabyte JSON Lines file where each line is an order: {"order_id":..., "user_id":..., "amount":..., "currency":..., "ts":...}. Compute, per user: total order count, sum of amounts (in USD — assume a static FX map), and last order timestamp. Output as JSON. The file is too large to load. Optimize for memory, then speed.

Clarify
  • How many distinct users — fits in memory?
  • What about malformed JSON lines? Skip or fail?
  • Are amounts in cents (integer) or decimals? (Float math on money is a footgun.)
  • Is the FX map provided, or do we need to fetch it?
Hints
  1. Use a streaming parser (Jackson's JsonParser) — never readTree for line-by-line.
  2. Use BigDecimal for money. Always.
  3. If user count is too large, consider an external sort by user_id + sequential aggregation, or a disk-backed map.
  4. For speed: parallelize. Split the file into chunks (handling line boundaries), aggregate each in a worker, merge per-user maps at the end.
Approach
▸ Design

Two scaling axes pull in opposite directions:

  • File size — multi-gigabyte. Solution: stream line-by-line, never load the whole file.
  • Per-line work — JSON parsing is the dominant CPU cost. Solution: use a streaming JSON parser (Jackson JsonParser in Java, simdjson or RapidJSON SAX in C++), not the convenient tree-builder (readTree, nlohmann::json::parse for whole documents).

The streaming parser walks tokens — START_OBJECT, FIELD_NAME, VALUE, END_OBJECT — and you extract just the four fields you need without materializing the whole DOM. For one-shot per-line use it's a smaller win, but it composes correctly when lines get fat.

Money math: never use double for currency. 0.1 + 0.2 ≠ 0.3 in IEEE-754, and accumulated rounding error in summed transaction amounts becomes audit-visible quickly. Use BigDecimal in Java, or a fixed-point integer type (microcents, etc.) in C++. The C++ example here uses double for brevity but flags the issue — be ready to defend that choice or swap it for boost::multiprecision::cpp_dec_float or a custom Decimal class.

Per-user state: a hash map keyed on user_id, accumulating count + sum + last-timestamp. Memory scales with distinct users, not events.

Parallelization is the hard follow-up. Memory-map the file, scan for newline boundaries near chunk edges (so workers don't split a JSON line), hand each chunk to a worker. Each worker builds its own per-user map; merge them at the end. Amdahl's law dictates the merge step matters at scale, so either keep maps small per worker or use a concurrent map shared by all workers and skip the merge.

If users don't fit in memory: external sort by user_id, then linear scan with running aggregations. Or push it into Spark / DuckDB / a real OLAP system — at that scale, hand-rolled is the wrong answer.

Solution
import com.fasterxml.jackson.core.*;
import com.fasterxml.jackson.databind.*;
import java.io.*;
import java.math.*;
import java.nio.file.*;
import java.time.*;
import java.util.*;

public class OrderAggregator {

    public static final class UserAgg {
        long count;
        BigDecimal totalUsd = BigDecimal.ZERO;
        Instant lastTs = Instant.EPOCH;

        void add(BigDecimal amountUsd, Instant ts) {
            count++;
            totalUsd = totalUsd.add(amountUsd);
            if (ts.isAfter(lastTs)) lastTs = ts;
        }
    }

    private final Map<String, BigDecimal> fxToUsd;
    private final JsonFactory jf = new JsonFactory();

    public OrderAggregator(Map<String, BigDecimal> fxToUsd) {
        this.fxToUsd = Map.copyOf(fxToUsd);
    }

    public Map<String, UserAgg> aggregate(Path file) throws IOException {
        Map<String, UserAgg> result = new HashMap<>();
        long bad = 0;

        try (BufferedReader r = Files.newBufferedReader(file)) {
            String line;
            while ((line = r.readLine()) != null) {
                if (line.isBlank()) continue;
                try {
                    processLine(line, result);
                } catch (Exception e) {
                    bad++;  // in real code, log with line number
                }
            }
        }
        if (bad > 0) System.err.printf("skipped %d bad lines%n", bad);
        return result;
    }

    private void processLine(String line, Map<String, UserAgg> out)
            throws IOException {
        String userId = null, currency = null;
        BigDecimal amount = null;
        Instant ts = null;

        try (JsonParser p = jf.createParser(line)) {
            if (p.nextToken() != JsonToken.START_OBJECT) throw new IOException("not obj");
            while (p.nextToken() != JsonToken.END_OBJECT) {
                String field = p.currentName();
                p.nextToken();
                switch (field) {
                    case "user_id"  -> userId = p.getValueAsString();
                    case "amount"   -> amount = p.getDecimalValue();
                    case "currency" -> currency = p.getValueAsString();
                    case "ts"       -> ts = Instant.parse(p.getValueAsString());
                    default           -> p.skipChildren();
                }
            }
        }
        if (userId == null || amount == null || currency == null || ts == null) {
            throw new IOException("missing field");
        }
        BigDecimal rate = fxToUsd.get(currency);
        if (rate == null) throw new IOException("unknown currency: " + currency);
        BigDecimal usd = amount.multiply(rate);
        out.computeIfAbsent(userId, k -> new UserAgg()).add(usd, ts);
    }
}

  • Why streaming parser, not the convenience tree-builder? A tree-builder (readTree, nlohmann::json::parse) builds a full DOM in memory per line. Fine for tiny payloads. At scale: more allocations, more pointer chasing, more GC pressure (Java) or more allocator fragmentation (C++). Streaming parsers walk tokens with constant memory regardless of payload size and skip uninteresting fields cheaply. simdjson goes further — SIMD-accelerated parsing, multiple GB/s on modern CPUs.
  • Why BigDecimal / fixed-point for money? 0.1 + 0.2 = 0.30000000000000004 in IEEE-754. Sum 10 million transactions and you have audit-visible drift. Two correct approaches: (a) BigDecimal (Java) — arbitrary-precision decimals, slower but exact. Set MathContext explicitly to avoid surprises with division. (b) Integer cents/microcents — store amounts as int64 in the smallest unit. Fast, exact, no allocation. Industry standard for high-volume systems. Floating-point is acceptable only when error-bound > precision needed (e.g., real-time approximate dashboards).
  • How do you parallelize over a single huge file? Memory-map the file (FileChannel.map in Java, mmap in C++). Compute N chunk boundaries near offset fileSize × i / N. Adjust each boundary forward to the next newline so JSON lines aren't split. Hand each chunk's (start, end) to a worker; each builds a per-user partial map. Final reduction merges them — concurrent map writes during the chunked phase are avoided, so workers are truly independent. ForkJoinPool in Java; std::async or tbb::parallel_for in C++.
  • What if user count exceeds memory? Spill aggregation state to disk. Two approaches: (a) External sort by user_id, then linear scan — each user's records arrive contiguously, so accumulation is constant-memory per user. (b) Disk-backed map — RocksDB, LMDB, or a sharded file structure. Both are well-trodden. At true scale, push the problem to Spark/DuckDB/Redshift/BigQuery — at some point hand-rolled is the wrong answer.
  • What if FX rates change over time and need historical accuracy? Store historical FX as a time-keyed structure (TreeMap<Instant, Map<Currency, Rate>>). Look up the rate as of the transaction's timestamp, not now. Accountants will complain otherwise. For high-precision needs, you also need to track which rate source you used — cross-currency reconciliation requires it.
  • How do you handle malformed JSON gracefully? The example skips bad lines and counts them. Production: log the line number and the parse error to a dead-letter sink, alert on rate-of-bad-lines, never silently drop. The "skip" pattern is wrong if these are paying customer transactions — you need provable accounting that no transaction was lost.

Tiny HTTP router

~45 min Service Medium
Prompt

Implement a route-matching component for a small HTTP framework. API: register(method, pattern, handler) and resolve(method, path) which returns the matched handler plus extracted path parameters. Patterns can include parameters like /users/:id/orders/:orderId and a wildcard suffix like /static/*. Method-mismatched routes should return a "method not allowed" signal, not "not found".

Clarify
  • Is order-of-registration the conflict resolution rule, or do more specific routes win?
  • Should the wildcard match zero or more path segments?
  • Are we expected to handle thousands of routes — does match speed matter?
  • Do path params allow slashes? (Default: no.)
Hints
  1. Trie of path segments. Each node has children for literal segments, a single param child, and an optional wildcard child.
  2. For "method not allowed": when path matches but method doesn't, you need to know the path matched. Track this distinctly from "no match."
  3. Match priority: literal > param > wildcard.
  4. Don't reach for regex unless asked. Trie is faster and clearer.
Approach
▸ Design

Three matching modes — literal segment, named parameter, wildcard suffix — with priority literal > param > wildcard. The natural data structure is a trie over path segments.

Each trie node has:

  • A map of literal-segment children (users, orders, ...).
  • An optional single param child (only one param name per position; :userId and :id at the same depth would conflict).
  • An optional wildcard with its own per-method handlers.
  • A per-method handler map for the route ending at this node.

Resolution walks the trie trying literal first, then param (binding the segment to the param name), then wildcard. The recursive walk backtracks if a branch dead-ends — important when, say, /users/admin is a literal route and /users/:id is a param route. Trying admin first as literal succeeds; trying admin as :id would also "succeed" but match the wrong route.

"Method not allowed" vs "not found" is what trips most candidates up. The HTTP spec is explicit: if the path matches but the method doesn't, return 405 with an Allow header listing valid methods. That requires distinguishing "no node matched the path" from "node matched but method missing." Both Java (sealed interface) and C++ (std::variant) make this clean — three case types in a sum type, exhaustive matching at the call site.

Concurrency: if routes are registered at startup and never change, the trie is effectively immutable after startup — no locks needed for resolution. If routes can change at runtime, copy-on-write the whole trie behind an atomic pointer swap; cheaper than locking and avoids reader stalls.

Why not regex? Real frameworks used to. Tries make priority obvious, allow O(path-length) matching, and avoid regex backtracking pathologies. Modern frameworks (Express, gin, Vert.x) have all converged on tries.

Solution
import java.util.*;
import java.util.function.*;

public class Router<H> {

    public sealed interface Resolution<H> {}
    public record Match<H>(H handler, Map<String, String> params)
        implements Resolution<H> {}
    public record MethodNotAllowed<H>(Set<String> allowed)
        implements Resolution<H> {}
    public record NotFound<H>() implements Resolution<H> {}

    private static final class Node<H> {
        Map<String, Node<H>> literal = new HashMap<>();
        String paramName;
        Node<H> paramChild;
        Map<String, H> wildcardHandlers;   // method -> handler at /*
        Map<String, H> handlers = new HashMap<>();
    }

    private final Node<H> root = new Node<>();

    public void register(String method, String pattern, H handler) {
        String[] segs = split(pattern);
        Node<H> node = root;
        for (int i = 0; i < segs.length; i++) {
            String s = segs[i];
            if (s.equals("*")) {
                if (i != segs.length - 1)
                    throw new IllegalArgumentException("* must be last");
                if (node.wildcardHandlers == null) node.wildcardHandlers = new HashMap<>();
                node.wildcardHandlers.put(method, handler);
                return;
            }
            if (s.startsWith(":")) {
                if (node.paramChild == null) {
                    node.paramChild = new Node<>();
                    node.paramName = s.substring(1);
                } else if (!node.paramName.equals(s.substring(1))) {
                    throw new IllegalArgumentException("conflicting param name");
                }
                node = node.paramChild;
            } else {
                node = node.literal.computeIfAbsent(s, k -> new Node<>());
            }
        }
        node.handlers.put(method, handler);
    }

    public Resolution<H> resolve(String method, String path) {
        String[] segs = split(path);
        Map<String, String> params = new HashMap<>();
        Node<H> matched = walk(root, segs, 0, params);
        if (matched == null) return new NotFound<>();
        H handler = matched.handlers.get(method);
        if (handler != null) return new Match<>(handler, params);
        if (!matched.handlers.isEmpty())
            return new MethodNotAllowed<>(Set.copyOf(matched.handlers.keySet()));
        return new NotFound<>();
    }

    private Node<H> walk(Node<H> node, String[] segs, int i,
                          Map<String, String> params) {
        if (i == segs.length) return node;

        // Try literal first (most specific)
        Node<H> lit = node.literal.get(segs[i]);
        if (lit != null) {
            Node<H> r = walk(lit, segs, i + 1, params);
            if (r != null) return r;
        }
        // Then param
        if (node.paramChild != null) {
            params.put(node.paramName, segs[i]);
            Node<H> r = walk(node.paramChild, segs, i + 1, params);
            if (r != null) return r;
            params.remove(node.paramName);
        }
        // Finally, wildcard at this node consumes everything remaining
        if (node.wildcardHandlers != null) {
            Node<H> pseudo = new Node<>();
            pseudo.handlers.putAll(node.wildcardHandlers);
            return pseudo;
        }
        return null;
    }

    private static String[] split(String path) {
        String p = path.startsWith("/") ? path.substring(1) : path;
        if (p.isEmpty()) return new String[0];
        return p.split("/");
    }
}
// C++17. Trie-based HTTP route matcher with sealed-style result.
#include <memory>
#include <optional>
#include <sstream>
#include <string>
#include <unordered_map>
#include <unordered_set>
#include <variant>
#include <vector>

template <class H>
class Router {
public:
    struct Match { H handler; std::unordered_map<std::string, std::string> params; };
    struct MethodNotAllowed { std::unordered_set<std::string> allowed; };
    struct NotFound {};
    using Resolution = std::variant<Match, MethodNotAllowed, NotFound>;

private:
    struct Node {
        std::unordered_map<std::string, std::unique_ptr<Node>> literal;
        std::string paramName;
        std::unique_ptr<Node> paramChild;
        std::optional<std::unordered_map<std::string, H>> wildcard;
        std::unordered_map<std::string, H> handlers;
    };
    Node root_;

    static std::vector<std::string> split(const std::string& path) {
        std::vector<std::string> out;
        std::stringstream ss(path[0] == '/' ? path.substr(1) : path);
        std::string seg;
        while (std::getline(ss, seg, '/')) if (!seg.empty()) out.push_back(seg);
        return out;
    }

    Node* walk(Node* node, const std::vector<std::string>& segs, size_t i,
               std::unordered_map<std::string, std::string>& params) {
        if (i == segs.size()) return node;

        auto lit = node->literal.find(segs[i]);
        if (lit != node->literal.end())
            if (auto* r = walk(lit->second.get(), segs, i + 1, params)) return r;

        if (node->paramChild) {
            params[node->paramName] = segs[i];
            if (auto* r = walk(node->paramChild.get(), segs, i + 1, params)) return r;
            params.erase(node->paramName);
        }
        if (node->wildcard) {
            static thread_local Node pseudo;
            pseudo.handlers = *node->wildcard;
            return &pseudo;
        }
        return nullptr;
    }

public:
    void registerRoute(const std::string& method,
                       const std::string& pattern, H handler) {
        auto segs = split(pattern);
        Node* node = &root_;
        for (size_t i = 0; i < segs.size(); ++i) {
            const auto& s = segs[i];
            if (s == "*") {
                if (i + 1 != segs.size())
                    throw std::invalid_argument("* must be last");
                if (!node->wildcard) node->wildcard.emplace();
                (*node->wildcard)[method] = std::move(handler);
                return;
            }
            if (s[0] == ':') {
                if (!node->paramChild) {
                    node->paramChild = std::make_unique<Node>();
                    node->paramName = s.substr(1);
                }
                node = node->paramChild.get();
            } else {
                auto& child = node->literal[s];
                if (!child) child = std::make_unique<Node>();
                node = child.get();
            }
        }
        node->handlers[method] = std::move(handler);
    }

    Resolution resolve(const std::string& method, const std::string& path) {
        auto segs = split(path);
        std::unordered_map<std::string, std::string> params;
        Node* m = walk(&root_, segs, 0, params);
        if (!m) return NotFound{};
        auto hIt = m->handlers.find(method);
        if (hIt != m->handlers.end()) return Match{hIt->second, params};
        if (!m->handlers.empty()) {
            std::unordered_set<std::string> allowed;
            for (auto& [k, _] : m->handlers) allowed.insert(k);
            return MethodNotAllowed{allowed};
        }
        return NotFound{};
    }
};
▸ Discussion / Follow-ups (with answers)
  • Why sealed types / variants for the result? Three distinct outcomes — Match, MethodNotAllowed, NotFound — each carrying different data. Encoding them as a sum type makes the call site exhaustive: switch on the result, handle every case, the compiler enforces it. The alternatives — nullable returns, status enums, exceptions — all bury the structure. Java 17's sealed interfaces and pattern matching make this clean; std::variant in C++ with std::visit is the equivalent.
  • How do you handle concurrent route registration? Two cases. (1) Static routes — registered once at startup, immutable thereafter. The trie is effectively const after init; no locks needed for the resolution path (which is the hot path). (2) Dynamic routes — copy-on-write. Build a new trie with the new route added, atomically swap the pointer. Old in-flight resolves continue on the old trie; new ones use the new one. Cheaper than locking, no reader stalls. AtomicReference in Java; std::atomic<std::shared_ptr> in C++.
  • How do you add wildcard binding (capture the suffix)? Express does this — /static/*path binds the matched suffix to path. Implementation: when the trie walks into a wildcard, capture remaining segments (joined by /) into a synthetic param. The wildcard node carries the param name. One-line addition to the trie node, two-line addition to the walker.
  • How do you support route priority other than literal > param > wildcard? Some frameworks allow explicit priority annotations. Implementation: track priority per route and use it as a tiebreaker when multiple paths match. Most don't — the literal > param > wildcard convention is intuitive and conflicts are rare in practice. If priorities must be configurable, gin and httprouter use a priority field on the trie node and resolve at registration time.
  • Why not regex per route? (1) Performance — regex matching is O(path-length) per route, so total is O(routes × path-length); trie is O(path-length) total. With hundreds of routes, the difference is real. (2) Reasoning — regex backtracking can be exponential on adversarial input (ReDoS). Tries can't backtrack pathologically. (3) Conflicts — trie structure makes route conflicts visible at registration time; regex order-of-registration is implicit and bug-prone. Regex was the old way; nobody who's tried both goes back.
  • How do you handle case-sensitivity, trailing slashes, query strings? Convention: paths are case-sensitive (per RFC 3986 for paths, though the standard is fuzzy in practice — most frameworks normalize). Trailing slash: either canonicalize to no trailing slash before matching, or treat /users and /users/ as distinct. Most frameworks redirect one to the other. Query strings: stripped before matching; passed through to the handler as a separate map.

Request validation pipeline

~35 min Service Warm-up
Prompt

Build a composable validation API. Given an inbound request payload (a Map<String, Object>), validate it against a set of rules — required fields, type checks, regex, range, custom predicates. Return all violations, not just the first. The API should let callers compose validators ergonomically.

Clarify
  • Field paths nested? (address.zip)
  • Should "required" prevent further checks on that field, or proceed?
  • I18n on error messages?
  • Is short-circuiting on first error ever desired?
Hints
  1. Each Validator takes the input and returns a List<Violation>. Composition = list-concatenation.
  2. A field-scoped DSL (field("email").required().matches(EMAIL_REGEX)) reads beautifully and tests cleanly.
  3. Don't use exceptions for control flow — they're slow and clunky for "we found 12 issues."
Approach
▸ Design

The choice that matters most: collect all errors, don't fail on the first one. Caller-side, "your email is wrong AND your age is missing AND your zip is malformed" is one round-trip and one fix. Failing fast forces three round-trips. UX wins — and APIs that do this correctly are a small differentiator.

The shape: a Validator is a function from request → list of violations. Composition is just list-concatenation: (a + b)(req) = a(req) ++ b(req). This makes validators trivially combinable with no class hierarchy.

Within a single field, build up checks via a field-scoped DSL: field("email").required().type(String).matches(EMAIL_REGEX). Reads naturally, tests cleanly, and each check is a small isolated function. The DSL is just a builder under the hood — required(), type(), etc. each push a check function onto an internal list.

Why functional, not annotation-based? Bean Validation (JSR-380) and equivalents are fine for static models, but they're inflexible — hard to compose runtime rules from config, hard to test in isolation, hard to use when the schema isn't a fixed POJO. Function-based DSLs work for any input source, including dynamic Maps as in this prompt.

Required vs optional fields: if a field is missing AND not marked required, skip its remaining checks. Otherwise you'd report "must be a string" on a field that wasn't supposed to be there. The required check is special — it's the only check that runs on a null/missing value.

Performance: validators are immutable once built. Build one per endpoint at startup; reuse across requests. No reflection, no allocation per call beyond the violation list.

Nested fields (address.zip): two approaches. (a) Flatten — walk the path with split-on-dot. (b) Recurse — sub-validators that operate on sub-objects. (b) is cleaner for deeply nested schemas; (a) is fine for one or two levels.

Solution
import java.util.*;
import java.util.function.*;
import java.util.regex.*;

public final class Validation {

    public record Violation(String field, String code, String message) {}

    public interface Validator extends Function<Map<String, Object>, List<Violation>> {

        default Validator and(Validator next) {
            return req -> {
                List<Violation> merged = new ArrayList<>(this.apply(req));
                merged.addAll(next.apply(req));
                return merged;
            };
        }
    }

    public static FieldRule field(String name) { return new FieldRule(name); }

    public static final class FieldRule implements Validator {
        private final String name;
        private final List<Function<Object, Optional<Violation>>> checks = new ArrayList<>();
        private boolean requiredCheck = false;

        FieldRule(String name) { this.name = name; }

        public FieldRule required() {
            requiredCheck = true;
            return add(v -> v == null
                ? Optional.of(new Violation(name, "REQUIRED", name + " is required"))
                : Optional.empty());
        }

        public FieldRule type(Class<?> t) {
            return add(v -> (v != null && !t.isInstance(v))
                ? Optional.of(new Violation(name, "TYPE",
                    name + " must be " + t.getSimpleName()))
                : Optional.empty());
        }

        public FieldRule matches(Pattern regex) {
            return add(v -> (v instanceof String s && !regex.matcher(s).matches())
                ? Optional.of(new Violation(name, "FORMAT", name + " format invalid"))
                : Optional.empty());
        }

        public FieldRule range(long min, long max) {
            return add(v -> {
                if (!(v instanceof Number n)) return Optional.empty();
                long x = n.longValue();
                return (x < min || x > max)
                    ? Optional.of(new Violation(name, "RANGE",
                        name + " must be in [" + min + "," + max + "]"))
                    : Optional.empty();
            });
        }

        public FieldRule satisfies(Predicate<Object> pred, String code, String msg) {
            return add(v -> (v != null && !pred.test(v))
                ? Optional.of(new Violation(name, code, msg))
                : Optional.empty());
        }

        private FieldRule add(Function<Object, Optional<Violation>> c) {
            checks.add(c); return this;
        }

        public List<Violation> apply(Map<String, Object> req) {
            Object value = req.get(name);
            List<Violation> out = new ArrayList<>();

            if (value == null && !requiredCheck) return out;
            for (var check : checks) {
                check.apply(value).ifPresent(out::add);
            }
            return out;
        }
    }
}

class FieldRule {
    std::string name_;
    std::vector<std::function<std::optional<Violation>(const std::any*)>> checks_;
    bool required_ = false;

        required_ = true;
        checks_.push_back([n = name_](const std::any* v) -> std::optional<Violation> {
            if (!v || !v->has_value()) return Violation{n, "REQUIRED", n + " is required"};
            return std::nullopt;
        });
        return *this;
    }

    template <class T>
    FieldRule& type() {
        checks_.push_back([n = name_](const std::any* v) -> std::optional<Violation> {
            if (!v || !v->has_value()) return std::nullopt;
            if (!std::any_cast<T>(v)) return Violation{n, "TYPE", n + " wrong type"};
            return std::nullopt;
        });
        return *this;
    }

    FieldRule& matches(std::regex re) {
        checks_.push_back([n = name_, re = std::move(re)](const std::any* v)
                -> std::optional<Violation> {
            if (!v || !v->has_value()) return std::nullopt;
            auto* s = std::any_cast<std::string>(v);
            if (s && !std::regex_match(*s, re))
                return Violation{n, "FORMAT", n + " format invalid"};
            return std::nullopt;
        });
        return *this;
    }

    operator Validator() const {
        auto name = name_;
        auto checks = checks_;
        auto required = required_;
        return [name, checks, required](const Payload& p) {
            std::vector<Violation> out;
            auto it = p.find(name);
            const std::any* v = (it != p.end()) ? &it->second : nullptr;
            if (!v && !required) return out;
            for (auto& c : checks) if (auto r = c(v)) out.push_back(*r);
            return out;
        };
    }
};

  • Why functional, not Bean Validation / annotation-based? Annotations are static — the rules live with the model class. Function-based composition gives you all four cleanly. Bean Validation is fine for "validate my User entity"; it's wrong for "validate this incoming JSON body against this endpoint's rules."

(b) Recurse — object("address", subValidator) applies a validator to a sub-map. Composes naturally for deep schemas. (b) is what JSON Schema validators do; (a) is what most ad-hoc DSLs do. Pick based on schema depth.
  • How do you internationalize error messages? Don't put text in violations. Put codes ("REQUIRED", "FORMAT.EMAIL") plus parameters ({"min": 0, "max": 150}). Default should be collect-all; short-circuit is the opt-in.

No reflection, no annotation scanning, no allocation per call beyond the violations list. Prompt

  • Should evaluation short-circuit on a BLOCK?
  • Are nested composite rules (any, all) needed? (Yes, in this case.)
  • Is config validation/loading separate from evaluation?
  • What's the priority when multiple rules fire on conflicting actions?
Hints
  1. Strategy pattern, registry-style. Map<String, RuleFactory> lookup keyed on the type field.
  2. Each rule implements boolean evaluate(Transaction). Composite rules all/any recurse.
  3. Adding a new rule type = one class + one registry entry. That's the test the interviewer will run.

Approach
▸ Design

The interviewer's actual test isn't whether you can evaluate the rules. It's whether adding a new rule type is a one-class change. If you write a giant switch on rule type and have to edit it for every new rule, that's the failing answer. The passing answer inverts the dependency: rules register themselves; the engine only knows about the registry.

This is the classic Strategy pattern with a registry. The pieces:

  • Rule — interface with an evaluate(transaction) method and an optional action.
  • RuleFactory — knows how to build a Rule from a JSON spec.
  • RuleRegistry — maps type-strings to factories. registry.build(spec) looks up the factory by spec.type and delegates.

Adding "velocity check"? Write a VelocityRule class and a factory, register it: registry.register("velocity", (n, r) -> new VelocityRule(...)). Engine code unchanged. That's the test.

Composite rules (all, any) are recursive — they hold a list of sub-rules and combine their results. They take the registry too, because they need to build their children. Notice that the recursion is open: composites can contain composites, naturally.

Action propagation: only top-level rules carry actions in this design. Inner rules of a composite don't fire actions independently — only the composite does, when all its children agree. This matches the JSON in the prompt and is what most real fraud engines do.

Validate at load time, not eval time. If a rule type is unknown, fail at config-load. The hot path (evaluate) should never throw on bad config.

Hot reload: build a new RuleEngine from new config, atomically swap the reference (volatile in Java, std::atomic<std::shared_ptr> in C++). Old in-flight evaluations finish on the old engine; new ones use the new one. No locking on the hot path.

public class RuleEngine {

    public record Transaction(String userId, BigDecimal amount,
                              String country, Instant accountCreated,
                              Instant ts) {}

    public interface Rule {
        boolean evaluate(Transaction tx);
        String action();   // FLAG / BLOCK / REVIEW / null for inner rules
    }

    public interface RuleFactory {
        Rule build(JsonNode spec, RuleRegistry registry);
    }

    public static final class RuleRegistry {
        private final Map<String, RuleFactory> factories = new HashMap<>();

        public RuleRegistry register(String type, RuleFactory factory) {
            factories.put(type, factory); return this;
        }

        public Rule build(JsonNode spec) {
            String type = spec.get("type").asText();
            RuleFactory f = factories.get(type);
            if (f == null) throw new IllegalArgumentException("unknown rule type: " + type);
            return f.build(spec, this);
        }
    }

    // ---- Built-in rules ---------------------------------------------------

    public static RuleRegistry defaultRegistry() {
        return new RuleRegistry()
            .register("amount_gt", (n, r) -> new PredicateRule(
                tx -> tx.amount().compareTo(new BigDecimal(n.get("value").asText())) > 0,
                actionOrNull(n)))
            .register("country_in", (n, r) -> {
                Set<String> set = new HashSet<>();
                n.get("value").forEach(v -> set.add(v.asText()));
                return new PredicateRule(tx -> set.contains(tx.country()), actionOrNull(n));
            })
            .register("new_account", (n, r) -> {
                long days = n.get("days").asLong();
                return new PredicateRule(
                    tx -> Duration.between(tx.accountCreated(), tx.ts()).toDays() <= days,
                    actionOrNull(n));
            })
            .register("all", (n, r) -> new CompositeRule(buildList(n, r), true, actionOrNull(n)))
            .register("any", (n, r) -> new CompositeRule(buildList(n, r), false, actionOrNull(n)));
    }

    private static List<Rule> buildList(JsonNode n, RuleRegistry r) {
        List<Rule> out = new ArrayList<>();
        n.get("rules").forEach(child -> out.add(r.build(child)));
        return out;
    }

    private static String actionOrNull(JsonNode n) {
        return n.has("action") ? n.get("action").asText() : null;
    }

    private record PredicateRule(Predicate<Transaction> pred, String action)
        implements Rule {
        public boolean evaluate(Transaction tx) { return pred.test(tx); }
    }

    private record CompositeRule(List<Rule> children, boolean all, String action)
        implements Rule {
        public boolean evaluate(Transaction tx) {
            return all
                ? children.stream().allMatch(c -> c.evaluate(tx))
                : children.stream().anyMatch(c -> c.evaluate(tx));
        }
    }

    // ---- Engine -----------------------------------------------------------

    private final List<Rule> rules;

    public RuleEngine(JsonNode config, RuleRegistry registry) {
        List<Rule> built = new ArrayList<>();
        config.get("rules").forEach(r -> built.add(registry.build(r)));
        this.rules = List.copyOf(built);
    }

    public List<String> evaluate(Transaction tx) {
        List<String> actions = new ArrayList<>();
        for (Rule r : rules) {
            if (r.evaluate(tx) && r.action() != null) actions.add(r.action());
        }
        return actions;
    }
}

        if (it == factories_.end())

    }

    std::unordered_map<std::string, Factory> factories_;

// Generic predicate-based rule
struct PredicateRule : Rule {
    std::function<bool(const Transaction&)> pred;
    bool evaluate(const Transaction& t) const override { return pred(t); }
};

struct CompositeRule : Rule {
    std::vector<std::unique_ptr<Rule>> children;
    bool all;
    bool evaluate(const Transaction& t) const override {
        if (all) {
            for (auto& c : children) if (!c->evaluate(t)) return false;
            return true;
        }
        for (auto& c : children) if (c->evaluate(t)) return true;
        return false;
    }
};

  • Why a registry, not a switch on rule type? The switch couples the engine code to every rule type — every new rule edits the engine. The registry inverts that: rules register themselves at startup; the engine knows nothing about specific types. Open-closed principle in practice. The interviewer's actual test is whether adding a new rule type is a one-class change. With the registry, it is.
  • How do you validate config at load time? Walk the JSON before the engine accepts it. For each rule node: check that the type exists in the registry, check that required fields are present (each factory can declare its required fields), recursively validate composite children. Run this once at startup; surface failures as exceptions before the service starts handling requests. The eval hot path should never throw on bad config.
  • How do you reload config at runtime without downtime? Build a new RuleEngine from the new config, then atomically swap the reference (volatile / AtomicReference in Java; std::atomic<std::shared_ptr> in C++). Old in-flight evaluations finish on the old engine — no shared state to corrupt — and new ones use the new engine. No locking on the hot path; no coordination needed.
  • How do you make this fast for very high QPS? Three optimizations. (1) Compile to predicate trees once — the design already does this; the registry builds a closure per rule at construction. (2) Short-circuit composites — all stops on first false, any stops on first true. Both built-in. (3) JIT / expression-tree compilation — for extreme cases, libraries like Janino (Java) or expression-template libraries (C++) can compile rules to bytecode/native code. Rarely worth it; the JIT typically optimizes well-shaped predicate trees just fine.

If you allowed inner actions, semantics get muddled — does an inner FLAG fire if the outer composite is any and another branch already triggered BLOCK? Better to keep it clean: only top-level rules carry actions.

Circuit breaker

~50 min Primitive Hard
Prompt

Implement a circuit breaker with three states: CLOSED (calls flow normally), OPEN (calls fail-fast), and HALF_OPEN (a few probe calls allowed). Trips to OPEN when failure rate exceeds a threshold over the last N calls. After a cool-down, transitions to HALF_OPEN; if probes succeed, returns to CLOSED. Thread-safe.

Clarify
  • Failure rate over a count window or a time window?
  • How many probes in HALF_OPEN? All-or-nothing, or a threshold?
  • What's a "failure"? Any exception, or only certain types (e.g. IOException)?
  • Should slow calls count as failures? (Resilience4j does.)
Hints
  1. Use atomic state transitions. The state is the source of truth; stats inform transitions but live separately.
  2. The OPEN→HALF_OPEN transition is time-based — check on each call rather than scheduling.
  3. In HALF_OPEN, atomically allow only N probes through; the rest fast-fail.
  4. Mention that real implementations (Resilience4j) batch event recording for performance.
Approach
▸ Design

Three states form a small state machine:

  • CLOSED — calls flow normally; failures are tallied.
  • OPEN — calls fail-fast immediately, without invoking the underlying operation. After a cool-down period, transitions to HALF_OPEN.
  • HALF_OPEN — a small number of "probe" calls are allowed through. If they succeed, the breaker closes again. If any fail, it re-opens.

The state is the source of truth. Statistics inform transitions but are kept separate. Each transition is implemented with compare-and-set rather than plain writes — two threads simultaneously deciding to trip the breaker would otherwise both write OPEN and corrupt the timer (each setting openedAt to its own now). CAS guarantees the transition happens exactly once; the loser thread observes the winner's transition and skips its own.

OPEN → HALF_OPEN transition is time-based. The cleanest implementation is to check elapsed time on each call, rather than scheduling a timer fire. If the call finds the breaker OPEN with elapsed > cooldown, it CASes to HALF_OPEN and proceeds as a probe. No background threads, no scheduler dependencies.

Probe limiting in HALF_OPEN: only allow N probes through; the rest fast-fail like OPEN. This is what prevents a single recovery attempt from being clobbered by full traffic. Implement with an atomic counter — increment-and-check, decrement on completion.

Failure counting via a fixed-size ring buffer. The "filled" guard is critical: don't trip on a single failure when the buffer's only seen one call. Without that guard, the breaker is hair-trigger on cold start.

What counts as a failure? Up to you. Standard answer: any thrown exception. Resilience4j extends this with slow-call detection — a call that succeeds but takes longer than threshold counts as a failure. Worth mentioning if asked.

The cleanup edge case: when HALF_OPEN succeeds and transitions back to CLOSED, reset the failure ring buffer. Otherwise old failures persist and could re-trip the breaker prematurely.

Solution
import java.time.*;
import java.util.concurrent.*;
import java.util.concurrent.atomic.*;

public class CircuitBreaker {

    public enum State { CLOSED, OPEN, HALF_OPEN }

    public static class CircuitOpenException extends RuntimeException {}

    public record Config(int windowSize, double failureThreshold,
                          Duration openDuration, int halfOpenProbes) {}

    private final Config cfg;
    private final AtomicReference<State> state = new AtomicReference<>(State.CLOSED);
    private final AtomicLong openedAt = new AtomicLong();
    private final AtomicInteger probesInFlight = new AtomicInteger();
    private final AtomicInteger probeSuccesses = new AtomicInteger();
    private final RingBuffer outcomes;

    public CircuitBreaker(Config cfg) {
        this.cfg = cfg;
        this.outcomes = new RingBuffer(cfg.windowSize());
    }

    public <T> T execute(Callable<T> op) throws Exception {
        State s = currentState();
        if (s == State.OPEN) throw new CircuitOpenException();
        if (s == State.HALF_OPEN) {
            if (probesInFlight.incrementAndGet() > cfg.halfOpenProbes()) {
                probesInFlight.decrementAndGet();
                throw new CircuitOpenException();
            }
        }
        try {
            T result = op.call();
            onSuccess(s);
            return result;
        } catch (Exception e) {
            onFailure(s);
            throw e;
        } finally {
            if (s == State.HALF_OPEN) probesInFlight.decrementAndGet();
        }
    }

    private State currentState() {
        State s = state.get();
        if (s == State.OPEN) {
            long elapsed = System.currentTimeMillis() - openedAt.get();
            if (elapsed >= cfg.openDuration().toMillis()
                    && state.compareAndSet(State.OPEN, State.HALF_OPEN)) {
                probesInFlight.set(0);
                probeSuccesses.set(0);
                return State.HALF_OPEN;
            }
        }
        return state.get();
    }

    private void onSuccess(State observed) {
        if (observed == State.HALF_OPEN) {
            if (probeSuccesses.incrementAndGet() >= cfg.halfOpenProbes()) {
                if (state.compareAndSet(State.HALF_OPEN, State.CLOSED)) {
                    outcomes.reset();
                }
            }
        } else {
            outcomes.record(true);
        }
    }

    private void onFailure(State observed) {
        if (observed == State.HALF_OPEN) {
            trip();
        } else {
            outcomes.record(false);
            if (outcomes.failureRate() >= cfg.failureThreshold()
                    && outcomes.filled()) {
                trip();
            }
        }
    }

    private void trip() {
        if (state.getAndSet(State.OPEN) != State.OPEN) {
            openedAt.set(System.currentTimeMillis());
        }
    }

    public State state() { return state.get(); }

    // Lock-free-ish ring buffer for outcomes. Coarse but adequate.
    private static final class RingBuffer {
        private final boolean[] buf;
        private final AtomicInteger idx = new AtomicInteger();
        private volatile int filled = 0;
        private final Object lock = new Object();

        RingBuffer(int size) { this.buf = new boolean[size]; }

        void record(boolean success) {
            synchronized (lock) {
                int i = idx.getAndIncrement() % buf.length;
                buf[i] = success;
                if (filled < buf.length) filled++;
            }
        }

        double failureRate() {
            synchronized (lock) {
                if (filled == 0) return 0.0;
                int fails = 0;
                for (int i = 0; i < filled; i++) if (!buf[i]) fails++;
                return (double) fails / filled;
            }
        }

        boolean filled() { return filled == buf.length; }

        void reset() { synchronized (lock) { idx.set(0); filled = 0; } }
    }
}

#include <atomic>
#include <chrono>
#include <exception>

#include <mutex>
#include <stdexcept>
#include <vector>

class CircuitOpenException : public std::runtime_error {
public:
    CircuitOpenException() : std::runtime_error("circuit open") {}
};

class CircuitBreaker {
public:
    enum class State { CLOSED, OPEN, HALF_OPEN };

    struct Config {
        int windowSize;
        double failureThreshold;
        std::chrono::milliseconds openDuration;
        int halfOpenProbes;
    };

private:
    Config cfg_;
    std::atomic<State> state_{State::CLOSED};
    std::atomic<int64_t> openedAtMs_{0};
    std::atomic<int> probesInFlight_{0};
    std::atomic<int> probeSuccesses_{0};

    std::vector<bool> outcomes_;
    int outcomesIdx_ = 0;
    int outcomesFilled_ = 0;
    std::mutex outcomesMu_;

    static int64_t nowMs() {
        return std::chrono::duration_cast<std::chrono::milliseconds>(
            std::chrono::steady_clock::now().time_since_epoch()).count();
    }

    void recordOutcome(bool success) {
        std::lock_guard<std::mutex> lock(outcomesMu_);
        outcomes_[outcomesIdx_++ % cfg_.windowSize] = success;
        if (outcomesFilled_ < cfg_.windowSize) outcomesFilled_++;
    }

    double failureRate() {
        std::lock_guard<std::mutex> lock(outcomesMu_);
        if (outcomesFilled_ == 0) return 0.0;
        int fails = 0;
        for (int i = 0; i < outcomesFilled_; ++i) if (!outcomes_[i]) fails++;
        return static_cast<double>(fails) / outcomesFilled_;
    }

    bool windowFilled() {
        std::lock_guard<std::mutex> lock(outcomesMu_);
        return outcomesFilled_ == cfg_.windowSize;
    }

    void trip() {
        State prev = state_.exchange(State::OPEN);
        if (prev != State::OPEN) openedAtMs_.store(nowMs());
    }

    State currentState() {
        State s = state_.load();
        if (s == State::OPEN &&
            nowMs() - openedAtMs_.load() >= cfg_.openDuration.count()) {
            State expected = State::OPEN;
            if (state_.compare_exchange_strong(expected, State::HALF_OPEN)) {
                probesInFlight_.store(0);
                probeSuccesses_.store(0);
                return State::HALF_OPEN;
            }
        }
        return state_.load();
    }

public:
    explicit CircuitBreaker(Config cfg) : cfg_(cfg), outcomes_(cfg.windowSize) {}

    template <class F>
    auto execute(F&& op) -> decltype(op()) {
        State s = currentState();
        if (s == State::OPEN) throw CircuitOpenException();
        if (s == State::HALF_OPEN) {
            if (probesInFlight_.fetch_add(1) + 1 > cfg_.halfOpenProbes) {
                probesInFlight_.fetch_sub(1);
                throw CircuitOpenException();
            }
        }
        struct Guard {
            CircuitBreaker* cb; State s;
            ~Guard() { if (s == State::HALF_OPEN) cb->probesInFlight_.fetch_sub(1); }
        } guard{this, s};

        try {
            auto result = op();
            if (s == State::HALF_OPEN) {
                if (probeSuccesses_.fetch_add(1) + 1 >= cfg_.halfOpenProbes) {
                    State expected = State::HALF_OPEN;
                    if (state_.compare_exchange_strong(expected, State::CLOSED)) {
                        std::lock_guard<std::mutex> lock(outcomesMu_);
                        outcomesIdx_ = 0; outcomesFilled_ = 0;
                    }
                }
            } else {
                recordOutcome(true);
            }
            return result;
        } catch (...) {
            if (s == State::HALF_OPEN) {
                trip();
            } else {
                recordOutcome(false);
                if (windowFilled() && failureRate() >= cfg_.failureThreshold) trip();
            }
            throw;
        }
    }

    State state() { return state_.load(); }
};
▸ Discussion / Follow-ups (with answers)
  • Why CAS for state transitions, not just volatile writes? volatile gives visibility — other threads see the update. CAS gives atomicity — the transition happens exactly once. Without CAS, two threads simultaneously deciding to trip the breaker both write OPEN and both write openedAt with their own now values; the timer is now wrong (the later write wins, so the cool-down is shortened). With CAS, one thread wins the transition, observes its own write, sets openedAt consistently. The other thread's CAS fails; it sees the winner's state and skips redundant work.
  • What's the difference between count-based and time-based windows for the failure ratio? Count-based (this example): "trip if ≥ 50% of last 100 calls failed." Simple, predictable memory, but tail latency matters less. Time-based: "trip if ≥ 50% of last 60 seconds of calls failed." More intuitive for ops, but needs bucketing (same structure as the sliding-window error-rate problem). Resilience4j supports both. Real services usually use time-based; count-based is simpler for interview code.
  • How do you detect slow calls as failures? Resilience4j's "slow call" feature: a call that succeeds but takes longer than a threshold counts as a failure for breaker purposes. Implementation: time the call, compare to threshold, record outcome accordingly. Useful when downstream is technically responding but at unusable latencies — your service is degrading even though no exceptions are thrown.
  • What's the production answer? Java: Resilience4j. It's the modern standard — modular, lightweight, integrates with reactive types and metrics. Hystrix is the older alternative; Netflix has deprecated it. C++: ecosystem is fragmented; you'd typically integrate with your service mesh (Envoy has circuit breaking built in) rather than handling it in-process. Mention the library; show you wouldn't ship hand-rolled breakers for production HTTP calls.
  • What if multiple HALF_OPEN probes are still in flight when one fails? The first failure trips the breaker back to OPEN. In-flight probes complete normally (no point cancelling them — they're already running) but their outcomes don't matter for state once OPEN is re-entered. The CAS guard on HALF_OPEN → OPEN ensures only the first failure's transition wins; subsequent failures see OPEN and don't re-execute the transition.
  • How do you expose state for observability? Expose the state, the failure rate, and the time-since-last-transition as metrics (via Micrometer / Prometheus). Operators want to see: which breakers are open, how long they've been open, what their failure rates were when they tripped. State changes should also emit events for audit. Resilience4j has this built in; if you're rolling your own, add a simple event listener interface.
  • How does this compose with retries? Carefully. Naive: retry inside breaker — each retry counts as a separate call, and a flaky downstream causes multiple failures per logical request, which trips the breaker faster than intended. Better: breaker outside retry — the breaker sees one logical attempt; the retry handles transient errors that don't deserve to count against the breaker's tally. Resilience4j composes them in this order via decorators.

In-memory job queue

~55 min Primitive Hard
Prompt

Build an in-memory job queue with submit(Job, priority, runAt) and a worker pool that pulls and runs jobs. Higher priority runs first; same priority runs in submission order. Jobs scheduled in the future shouldn't be picked up until their runAt time. Failed jobs retry up to N times with backoff. Multiple workers pull concurrently without dropping or duplicating jobs.

Clarify
  • Is durability required, or in-memory only? (In-memory per the prompt.)
  • What about job cancellation?
  • Should retries respect the original priority, or be deprioritized?
  • Is there a max queue size — and what happens when full?
Hints
  1. DelayQueue from java.util.concurrent handles "pickup after time T" beautifully.
  2. For priority + FIFO tiebreak: combine priority with a monotonic submission sequence in the comparator.
  3. Workers are just N threads in a loop calling queue.take().
  4. Reject the temptation to combine priority queue + scheduled executor manually. DelayQueue with a custom Delayed does both.
Approach
▸ Design

Three requirements layered on a simple priority queue:

  1. Priority — higher priority runs first.
  2. Delayed execution — jobs scheduled in the future shouldn't run until their time arrives.
  3. FIFO within priority — equal-priority jobs run in submission order.

Java has DelayQueue built in: a blocking queue where take() only returns elements whose delay has expired. Combined with a custom Delayed implementation that defines the priority and FIFO comparator, this gets all three for free. C++ has no equivalent, so the example builds it from std::priority_queue + std::condition_variable::wait_until. Same idea, more code.

The comparator is the trick. Three keys in priority order:

  1. Ready time, ascending — a task ready 5 minutes from now beats one ready 10 minutes from now.
  2. Priority, descending — among ready tasks, higher priority first.
  3. Submission sequence, ascending — among same-priority tasks, FIFO.

The submission sequence comes from a monotonic counter (AtomicLong in Java, std::atomic<int64_t> in C++) incremented on every submission. Without it, "same priority" ties resolve nondeterministically.

Why DelayQueue over a manual priority queue + sleep loop? The manual version has a classic bug: workers sleep until the next task's ready-time, but if a higher-priority task arrives during the sleep, you don't wake up. DelayQueue's blocking take() handles wake-on-insert correctly. condition_variable::wait_until in C++ does the same — workers block on the CV; a new submission notifies and they re-evaluate.

Retries: on failure, increment the attempt counter, compute backoff, requeue with a new ready time. The worker doesn't busy-loop or block — it just re-submits and goes back to take().

Daemon threads (Java) so the JVM can exit cleanly without explicit shutdown. C++ requires explicit join, so the destructor calls close() which sets running = false, notifies all workers, and joins them.

The persistence question is the natural follow-up. In-memory means jobs vanish on crash. Real systems (Sidekiq, Resque, RabbitMQ delayed plugin, AWS SQS) use external storage. The architecture is the same — the queue interface is the abstraction; swap the implementation.

Solution
import java.time.*;
import java.util.concurrent.*;
import java.util.concurrent.atomic.*;

public class JobQueue implements AutoCloseable {

    public interface Job { void run() throws Exception; }

    private static final AtomicLong SEQ = new AtomicLong();

    private static final class Task implements Delayed {
        final Job job;
        final int priority;             // higher = sooner
        final long seq;                 // FIFO tiebreak
        long readyAtNanos;
        int attempt;
        final int maxAttempts;

        Task(Job j, int pri, Instant runAt, int maxAttempts) {
            this.job = j;
            this.priority = pri;
            this.seq = SEQ.getAndIncrement();
            this.readyAtNanos = toNanos(runAt);
            this.maxAttempts = maxAttempts;
        }

        public long getDelay(TimeUnit unit) {
            return unit.convert(readyAtNanos - System.nanoTime(), TimeUnit.NANOSECONDS);
        }

        public int compareTo(Delayed o) {
            Task other = (Task) o;
            // 1) ready-time; 2) priority desc; 3) seq asc
            int c = Long.compare(this.readyAtNanos, other.readyAtNanos);
            if (c != 0) return c;
            c = Integer.compare(other.priority, this.priority);
            if (c != 0) return c;
            return Long.compare(this.seq, other.seq);
        }

        static long toNanos(Instant when) {
            long deltaMs = when.toEpochMilli() - System.currentTimeMillis();
            return System.nanoTime() + TimeUnit.MILLISECONDS.toNanos(Math.max(0, deltaMs));
        }
    }

    private final DelayQueue<Task> queue = new DelayQueue<>();
    private final Thread[] workers;
    private volatile boolean running = true;

    public JobQueue(int workerCount) {
        workers = new Thread[workerCount];
        for (int i = 0; i < workerCount; i++) {
            workers[i] = new Thread(this::runLoop, "jobworker-" + i);
            workers[i].setDaemon(true);
            workers[i].start();
        }
    }

    public void submit(Job j, int priority, Instant runAt, int maxAttempts) {
        queue.offer(new Task(j, priority, runAt, maxAttempts));
    }

    public void submit(Job j) { submit(j, 0, Instant.now(), 3); }

    private void runLoop() {
        while (running) {
            try {
                Task t = queue.take();
                try {
                    t.job.run();
                } catch (Exception e) {
                    t.attempt++;
                    if (t.attempt < t.maxAttempts) {
                        // requeue with exponential backoff
                        long backoffMs = 100L * (1L << t.attempt);
                        t.readyAtNanos = System.nanoTime()
                            + TimeUnit.MILLISECONDS.toNanos(backoffMs);
                        queue.offer(t);
                    } else {
                        System.err.printf("job exhausted retries: %s%n", e);
                    }
                }
            } catch (InterruptedException ie) {
                Thread.currentThread().interrupt();
                return;
            }
        }
    }

    public void close() throws InterruptedException {
        running = false;
        for (Thread w : workers) w.interrupt();
        for (Thread w : workers) w.join(1000);
    }
}
// C++17. In-memory job queue with priority + delayed execution + retry.
#include <atomic>
#include <chrono>
#include <condition_variable>
#include <functional>
#include <iostream>
#include <memory>
#include <mutex>
#include <queue>
#include <thread>
#include <vector>

class JobQueue {
    struct Task {
        std::function<void()> job;
        int priority;                 // higher = sooner
        int64_t seq;                  // FIFO tiebreak
        std::chrono::steady_clock::time_point readyAt;
        int attempt;
        int maxAttempts;
    };

    struct Cmp {
        bool operator()(const Task& a, const Task& b) const {
            // priority_queue is a max-heap; we want the *least* element on top.
            if (a.readyAt != b.readyAt) return a.readyAt > b.readyAt;
            if (a.priority != b.priority) return a.priority < b.priority;
            return a.seq > b.seq;
        }
    };

    std::priority_queue<Task, std::vector<Task>, Cmp> pq_;
    std::mutex mu_;
    std::condition_variable cv_;
    std::atomic<bool> running_{true};
    std::atomic<int64_t> nextSeq_{0};
    std::vector<std::thread> workers_;

    void runLoop() {
        while (running_.load()) {
            std::unique_lock<std::mutex> lock(mu_);
            cv_.wait(lock, [&] { return !running_ || !pq_.empty(); });
            if (!running_) return;

            auto now = std::chrono::steady_clock::now();
            if (pq_.top().readyAt > now) {
                cv_.wait_until(lock, pq_.top().readyAt);
                continue;
            }
            Task t = pq_.top();
            pq_.pop();
            lock.unlock();

            try {
                t.job();
            } catch (const std::exception& e) {
                t.attempt++;
                if (t.attempt < t.maxAttempts) {
                    int64_t backoff = 100LL * (1LL << std::min(t.attempt, 10));
                    t.readyAt = std::chrono::steady_clock::now() +
                                std::chrono::milliseconds(backoff);
                    {
                        std::lock_guard<std::mutex> lk(mu_);
                        pq_.push(std::move(t));
                    }
                    cv_.notify_one();
                } else {
                    std::cerr << "job exhausted retries: " << e.what() << '\n';
                }
            }
        }
    }

public:
    explicit JobQueue(int workerCount) {
        for (int i = 0; i < workerCount; ++i)
            workers_.emplace_back([this] { runLoop(); });
    }

    ~JobQueue() { close(); }

    void submit(std::function<void()> job, int priority = 0,
                std::chrono::steady_clock::time_point runAt =
                    std::chrono::steady_clock::now(),
                int maxAttempts = 3) {
        Task t{std::move(job), priority, nextSeq_.fetch_add(1),
               runAt, 0, maxAttempts};
        {
            std::lock_guard<std::mutex> lock(mu_);
            pq_.push(std::move(t));
        }
        cv_.notify_one();
    }

    void close() {
        if (!running_.exchange(false)) return;
        cv_.notify_all();
        for (auto& w : workers_) if (w.joinable()) w.join();
    }
};
▸ Discussion / Follow-ups (with answers)
  • How do you make this durable (survive crashes)? Outside the prompt, but the design extends cleanly. Two paths: (a) Write-ahead log — append every submit to a log file before returning; replay on startup to rebuild the queue. Mark jobs done in the log on completion. (b) External broker — push the queue out of process: RabbitMQ delayed plugin, AWS SQS with delay seconds, Redis sorted sets keyed on ready-time, or a real job queue (Sidekiq, BullMQ, dramatiq). The in-memory queue interface is a clean abstraction; the implementation is swap-in.
  • How do you handle backpressure when the queue grows unbounded? Two strategies. (a) Bounded queue — submit blocks (or rejects) when full. Backpressure flows back to producers naturally. (b) Spillover — overflow to disk or a slower tier. Most systems pick (a) — explicit failure beats silent OOM. Configure: BlockingJobQueue(capacity=10000, rejectionPolicy=BLOCK | DROP_OLDEST | THROW).
  • How do you support cancellation? Return a handle from submit with a cancel() method. Two implementations. (a) Best-effort — cancel() tries to remove the task from the queue; if it's already running, it sets a flag the job checks periodically. (b) Hard-cancel — running jobs are interrupted (Java Thread.interrupt) or their associated Future is cancelled with mayInterruptIfRunning=true. (a) is safer; (b) requires jobs to be interrupt-safe.
  • How do retries interact with priority? By default, retried jobs keep their original priority — fairness within the priority tier. Some systems demote retries (a job that's failed once is "less trusted"; let fresh jobs go first). Implement either way: in the Task, either reuse the original priority or subtract per-retry. Consider: do you want a permanently-failing high-priority job to monopolize the queue? Demoting retries prevents that.
  • What's a poison-pill job and how do you handle it? A job that always fails (bad input, fundamentally broken). Without a max-attempts cap, it retries forever. With one (this example uses maxAttempts), it eventually moves to a dead-letter sink — log it, alert on it, possibly move to a separate queue for manual inspection. The DLQ pattern is standard; it's the difference between a job queue that survives bad jobs and one that's clogged by them.
  • How do you parallelize beyond N worker threads? The single-machine ceiling is roughly N × (1 / per-job-time). Past that: distribute. Run multiple instances of this service, each pulling from a shared queue (Redis / Kafka / SQS). Each instance has its local worker pool. Job scheduling per machine is unchanged; the queue is what scales out. This is the architecture every real job system uses — Sidekiq, Celery, etc.
  • What's a sample bug in this code worth pointing out? The C++ comparator must produce a strict weak ordering. Three keys — readyAt, priority, seq — with consistent direction. The example handles this carefully (priority is the only descending key; readyAt and seq are ascending). Get the direction wrong on any key and std::priority_queue ordering breaks subtly — jobs get out of order without crashing, hardest kind of bug to find.
