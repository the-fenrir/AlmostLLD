Pricing engine for an e-commerce checkout

~30 min Composition Warm-Up
Prompt

You're modeling pricing for an online checkout. A cart has line items, a customer tier, and a shipping country. The final price is built from the subtotal by applying any number of price modifiers in order: percentage discounts, tier discounts, fixed shipping fees, taxes (which depend on country and apply after shipping in EU jurisdictions). The marketing team will keep adding new types of discounts. Build something that lets them.

The interviewer will then say: "now add a buy-one-get-one-free rule, then a coupon code rule, then a member-anniversary discount." If your design needs touching for each one, that's the failing answer.

Clarify
  • Are modifiers always applied in a fixed order, or is order configurable per checkout flow?
  • Can the same modifier appear twice (two stacking discounts)?
  • Is the breakdown (each step's contribution) needed for the receipt, or just the final total?
  • Are there modifiers that depend on running total being above/below a threshold?
Approach
▸ Design thinking

The instinct on first reading is "I'll add an enum of discount types and a switch in the pricing function." That's the wrong instinct, because the marketing team will be back next week with a new discount type and you'll be back in the same code adding a new case. The price you pay every time is: edit existing tested code, risk regressing existing discounts, re-test everything.

The right instinct: model each kind of price modification as a small object that knows its own rules. The pricing function holds a list of these objects and walks through them. Adding a new modifier is adding a new class; existing modifiers are untouched. This is the Open/Closed Principle in concrete form.

The structure has three pieces:

  • A PriceModifier interface — single method, takes the cart and the running total, returns the new running total.
  • Concrete modifiers — PercentageDiscount, TierDiscount, FlatShipping, VAT. Each ~10 lines. Each independently testable.
  • A Pipeline that holds an ordered list and applies them in sequence.

Order matters and order is data, not code. Discounts before shipping, shipping before tax — the pipeline configuration encodes this. Different countries, different markets, different checkout flows: each gets its own pipeline configuration. The pricing engine itself is the same.

Why BigDecimal / fixed-point cents: 0.1 + 0.2 ≠ 0.3 in IEEE-754 floats. For money, this is not negotiable. The interviewer might not call this out; bring it up yourself. Mentioning it unprompted signals that you've worked on production systems where floating-point money has burned someone.

Solution
// Pricing engine: each modifier is a step in a pipeline.
import java.math.BigDecimal;
import java.math.RoundingMode;
import java.util.ArrayList;
import java.util.List;

public final class Pricing {

    public record LineItem(String sku, BigDecimal unitPrice, int quantity) {
        public BigDecimal subtotal() {
            return unitPrice.multiply(BigDecimal.valueOf(quantity));
        }
    }

    public record Cart(List<LineItem> items, String customerTier, String shippingCountry) {
        public BigDecimal subtotal() {
            return items.stream().map(LineItem::subtotal)
                .reduce(BigDecimal.ZERO, BigDecimal::add);
        }
    }

    // Each modifier is a step: (cart, runningTotal) -> runningTotal.
    // Composition is just a list — no inheritance trees, no decorator wrapping.
    public interface PriceModifier {
        BigDecimal apply(Cart cart, BigDecimal running);
        String label();
    }

    public static final class PercentageDiscount implements PriceModifier {
        private final BigDecimal pct;
        private final String reason;
        public PercentageDiscount(double pct, String reason) {
            this.pct = BigDecimal.valueOf(pct);
            this.reason = reason;
        }
        public BigDecimal apply(Cart c, BigDecimal r) {
            return r.subtract(r.multiply(pct).divide(
                BigDecimal.valueOf(100), 2, RoundingMode.HALF_UP));
        }
        public String label() { return reason; }
    }

    public static final class TierDiscount implements PriceModifier {
        public BigDecimal apply(Cart c, BigDecimal r) {
            return switch (c.customerTier()) {
                case "gold"     -> r.subtract(r.multiply(BigDecimal.valueOf(0.10)));
                case "platinum" -> r.subtract(r.multiply(BigDecimal.valueOf(0.15)));
                default         -> r;
            };
        }
        public String label() { return "tier discount"; }
    }

    public static final class FlatShipping implements PriceModifier {
        private final BigDecimal fee;
        public FlatShipping(BigDecimal fee) { this.fee = fee; }
        public BigDecimal apply(Cart c, BigDecimal r) { return r.add(fee); }
        public String label() { return "shipping"; }
    }

    public static final class VAT implements PriceModifier {
        private final BigDecimal rate;
        public VAT(double rate) { this.rate = BigDecimal.valueOf(rate); }
        public BigDecimal apply(Cart c, BigDecimal r) {
            // EU VAT applies on the running total INCLUDING shipping
            if (c.shippingCountry().equals("DE") || c.shippingCountry().equals("FR"))
                return r.add(r.multiply(rate).divide(
                    BigDecimal.valueOf(100), 2, RoundingMode.HALF_UP));
            return r;
        }
        public String label() { return "VAT"; }
    }

    // The pipeline. Order matters — order is data, not control flow.
    public static final class Pipeline {
        private final List<PriceModifier> steps = new ArrayList<>();
        public Pipeline add(PriceModifier s) { steps.add(s); return this; }
        public BigDecimal price(Cart cart) {
            BigDecimal running = cart.subtotal();
            for (PriceModifier s : steps) running = s.apply(cart, running);
            return running;
        }
    }
}

#include <cstdint>

// Money as integer cents — never use double for currency.
struct Money {
    int64_t cents;
    static Money of(double dollars) {
        return {static_cast<int64_t>(dollars * 100 + (dollars >= 0 ? 0.5 : -0.5))};
    }
    Money operator+(Money o) const { return {cents + o.cents}; }
    Money operator-(Money o) const { return {cents - o.cents}; }
    Money percent(double p) const { return {static_cast<int64_t>(cents * p / 100.0)}; }
};

struct LineItem {
    std::string sku;
    Money unitPrice;
    int quantity;
    Money subtotal() const { return {unitPrice.cents * quantity}; }
};

struct Cart {
    std::vector<LineItem> items;
    std::string customerTier;       // "standard", "gold", etc.
    std::string shippingCountry;
    Money subtotal() const {
        Money total{0};
        for (auto& i : items) total = total + i.subtotal();
        return total;
    }
};

// PriceModifier is a step in the pricing pipeline. Each step transforms
// (cart, runningTotal) -> runningTotal. Composition is by listing steps;
// no inheritance trees, no decorator wrapping.
class PriceModifier {
public:
    virtual ~PriceModifier() = default;
    virtual Money apply(const Cart& cart, Money running) const = 0;
    virtual std::string label() const = 0;
};

class PercentageDiscount : public PriceModifier {
    double pct_;
    std::string reason_;
public:
    PercentageDiscount(double p, std::string reason)
        : pct_(p), reason_(std::move(reason)) {}
    Money apply(const Cart&, Money running) const override {
        return running - running.percent(pct_);
    }
    std::string label() const override { return reason_; }
};

class TierDiscount : public PriceModifier {
public:
    Money apply(const Cart& cart, Money running) const override {
        if (cart.customerTier == "gold") return running - running.percent(10);
        if (cart.customerTier == "platinum") return running - running.percent(15);
        return running;
    }
    std::string label() const override { return "tier discount"; }
};

class FlatShipping : public PriceModifier {
    Money fee_;
public:
    explicit FlatShipping(Money f) : fee_(f) {}
    Money apply(const Cart&, Money running) const override { return running + fee_; }
    std::string label() const override { return "shipping"; }
};

class VAT : public PriceModifier {
    double rate_;
public:
    explicit VAT(double r) : rate_(r) {}
    Money apply(const Cart& cart, Money running) const override {
        // EU VAT: applies on running total including shipping
        if (cart.shippingCountry == "DE" || cart.shippingCountry == "FR")
            return running + running.percent(rate_);
        return running;
    }
    std::string label() const override { return "VAT"; }
};

// The pricing pipeline. Order of modifiers is meaningful — discounts before
// shipping, shipping before tax (in most jurisdictions). The pipeline is
// data, not control flow.
class PricingPipeline {
    std::vector<std::shared_ptr<PriceModifier>> steps_;
public:
    PricingPipeline& add(std::shared_ptr<PriceModifier> step) {
        steps_.push_back(std::move(step));
        return *this;
    }
    Money price(const Cart& cart) const {
        Money running = cart.subtotal();
        for (auto& step : steps_) {
            Money before = running;
            running = step->apply(cart, running);
            (void)before;  // for tracing/breakdown
        }
        return running;
    }
};

int main() {
    Cart cart;
    cart.items = {
        {"SKU-1", Money::of(29.99), 2},
        {"SKU-2", Money::of(15.00), 1},
    };
    cart.customerTier = "gold";
    cart.shippingCountry = "DE";

    PricingPipeline p;
    p.add(std::make_shared<PercentageDiscount>(5, "promo"))
     .add(std::make_shared<TierDiscount>())
     .add(std::make_shared<FlatShipping>(Money::of(8.50)))
     .add(std::make_shared<VAT>(19));

    auto total = p.price(cart);
    std::cout << "total cents: " << total.cents << "\n";
    return 0;
}
▸ Discussion / Follow-ups (with answers)
  • How do you handle conditional modifiers — "10% off if subtotal > $50"? Either: (a) the modifier itself checks the condition and returns the running total unchanged if the condition isn't met, or (b) modifiers have a separate appliesTo(cart) method that the pipeline checks before calling apply. The first keeps the interface simpler; the second makes the rule visible to other tooling (e.g., a UI that shows "you're $5 away from a discount").
  • How do you compute a breakdown for the receipt? Have the pipeline track the running total before and after each step, returning a list of (modifier label, delta) alongside the final total. Two options: track the diff at each step, or have PriceModifier return a richer result type that includes the change. The diff approach keeps the modifier interface unchanged.
  • What about modifiers that need the cart's full state, not just the running total? The interface already passes the cart. A "free shipping if cart contains item X" modifier looks at cart.items(), not just the running total. The signature (cart, running) -> running handles this case naturally.
  • Are pipelines composable? Yes. A Pipeline can implement PriceModifier itself (its apply just runs its inner steps). This lets you say "the EU pipeline is the standard pipeline plus VAT at the end" without copying steps. This is the Composite pattern showing up naturally — the same shape as the rule engine in the previous practice set.
  • How do you handle currency? Use a Money type (see problem 11). The pipeline returns Money throughout; modifiers operate on Money. Don't mix Money and BigDecimal — the type system should prevent currency-blind arithmetic.
  • What if the marketing team wants a UI for editing pipelines? The pipeline is data; persist it as JSON. Each modifier serializes to {"type": "percentage_discount", "pct": 10, "reason": "promo"}. A UI builds the JSON; the engine reads it and instantiates modifiers via a registry. Strategy + registry — same shape as that problem.

Notification system with channels

~40 min Composition Medium
Prompt

You're building an alerting system for an SRE platform. Notifications go out via email, SMS, and push. Some channels need retries on failure; some need rate limiting; all need logging. Different alert priorities use different channel combinations — a P1 might use SMS+email, a P3 might use email only.

The interviewer will then ask: "now add a Slack channel," "now add a circuit breaker for the email provider," "now make the retries log how many tries it took." If those changes touch every channel implementation, your design is wrong.

Clarify
  • Should retries and rate limits be cross-cutting (apply to all channels) or per-channel?
  • Can the same notification go to multiple channels for one alert?
  • Are retries channel-specific (different policy per channel) or global?
  • What's the failure mode if all channels fail — propagate up or swallow and log?
Approach
▸ Design thinking

Two patterns at work, both worth being able to name and recognize:

Strategy pattern: each channel (Email, SMS, Push) is a different implementation of the same abstract operation (send a notification). The caller picks one based on requirements. The strategy is the channel itself; the abstraction is the Channel interface.

Decorator pattern: cross-cutting behaviors (retry, rate limiting, logging) are themselves Channels that wrap another Channel. A RetryingChannel contains another Channel and adds retry behavior. A RateLimitedChannel wraps and adds rate limiting. They compose: RateLimited(Retrying(Logging(Email))).

The signal that you're building a decorator: it implements the same interface as the thing it wraps, and it has a reference to one. is-a Channel + has-a Channel. You build behavior in layers; each layer is independently testable; new behaviors don't touch existing code.

What this design buys you on the follow-ups:

  • "Add a Slack channel" — write SlackChannel implements Channel. Done. Existing channels and decorators untouched.
  • "Add a circuit breaker" — write CircuitBreakerChannel as a new decorator. Wrap whichever channels need it. Existing code untouched.
  • "Make retries log attempts" — wrap differently: Logging(Retrying(Email)) instead of Retrying(Logging(Email)). Order changes the observability. No code changes.

Trap to avoid: putting retry/rate-limit/logging logic inside each channel's send method. That's the design that fails as soon as the second cross-cutting concern shows up — every channel duplicates it.

Solution
// Notification: strategy + decorator.
public final class Notifications {

    public record Notification(String recipient, String subject, String body) {}

    // Strategy: how to send. Different channels are different strategies.
    public interface Channel {
        void send(Notification n);
        String name();
    }

    public static final class EmailChannel implements Channel {
        public void send(Notification n) {
            System.out.printf("[email] -> %s: %s%n", n.recipient(), n.subject());
        }
        public String name() { return "email"; }
    }

    public static final class SmsChannel implements Channel {
        public void send(Notification n) {
            System.out.printf("[sms] -> %s: %s%n", n.recipient(), n.body());
        }
        public String name() { return "sms"; }
    }

    // Decorator: a Channel that wraps a Channel and adds behavior.
    // "is-a Channel + has-a Channel" — the key signal of decorator.
    public static final class RetryingChannel implements Channel {
        private final Channel inner;
        private final int maxAttempts;
        public RetryingChannel(Channel inner, int maxAttempts) {
            this.inner = inner; this.maxAttempts = maxAttempts;
        }
        public void send(Notification n) {
            for (int i = 0; i < maxAttempts; i++) {
                try { inner.send(n); return; }
                catch (RuntimeException e) {
                    if (i == maxAttempts - 1) throw e;
                    sleep(50L * (1L << i));
                }
            }
        }
        public String name() { return "retry(" + inner.name() + ")"; }

        private static void sleep(long ms) {
            try { Thread.sleep(ms); }
            catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        }
    }

    public static final class LoggingChannel implements Channel {
        private final Channel inner;
        public LoggingChannel(Channel inner) { this.inner = inner; }
        public void send(Notification n) {
            long start = System.nanoTime();
            inner.send(n);
            long us = (System.nanoTime() - start) / 1_000;
            System.out.printf("  [log] %s sent in %dus%n", inner.name(), us);
        }
        public String name() { return "logged(" + inner.name() + ")"; }
    }

    // Compose: log + retry + email
    public static Channel buildChannel() {
        return new LoggingChannel(new RetryingChannel(new EmailChannel(), 3));
    }
}

// Decorator: a Channel that wraps another Channel and adds behavior.
// Notice the "is-a Channel + has-a Channel" structure.
class RetryingChannel : public Channel {
    std::shared_ptr<Channel> inner_;
    int maxAttempts_;
public:
    RetryingChannel(std::shared_ptr<Channel> inner, int maxAttempts)
        : inner_(std::move(inner)), maxAttempts_(maxAttempts) {}
    void send(const Notification& n) override {
        for (int i = 0; i < maxAttempts_; ++i) {
            try { inner_->send(n); return; }
            catch (...) {
                if (i == maxAttempts_ - 1) throw;
                std::this_thread::sleep_for(std::chrono::milliseconds(50 * (1 << i)));
            }
        }
    }
    std::string name() const override { return "retry(" + inner_->name() + ")"; }
};

class LoggingChannel : public Channel {
    std::shared_ptr<Channel> inner_;
public:
    explicit LoggingChannel(std::shared_ptr<Channel> inner) : inner_(std::move(inner)) {}
    void send(const Notification& n) override {
        auto start = std::chrono::steady_clock::now();
        inner_->send(n);
        auto us = std::chrono::duration_cast<std::chrono::microseconds>(
            std::chrono::steady_clock::now() - start).count();
        std::cout << "  [log] " << inner_->name() << " sent in " << us << "us\n";
    }
    std::string name() const override { return "logged(" + inner_->name() + ")"; }
};

▸ Discussion / Follow-ups (with answers)
  • How do you handle "send to multiple channels for a single alert"? Wrap a list of channels in a BroadcastChannel that implements Channel and forwards to all. Failure semantics need a choice: all-or-nothing (any failure aborts), best-effort (failures logged, others continue), or at-least-one (succeed if any succeeds). State the policy on the broadcast channel.
  • How do you add per-channel rate limits? Wrap with a RateLimitedChannel decorator that owns a token bucket. Different channels can have different limits without coupling. Same shape as retry: decorator wraps and modulates.
  • Where does the "P1 uses email+SMS, P3 uses email only" routing live? Outside the channels. A NotificationDispatcher takes a priority and the notification, picks channels based on policy. Channels stay dumb. Routing logic is its own thing — testable without channel involvement.
  • How do you handle channel-specific failure (email provider is down)? Circuit breaker decorator. When the email provider fails repeatedly, the breaker opens and fast-fails. Fallback strategies — fall back to SMS if email circuit is open — live in the dispatcher, not the channels.
  • What about idempotency? A retry of "send notification" shouldn't deliver twice. Two layers. (a) Generate an idempotency key per logical notification; pass it to the channel; the channel forwards it to the underlying provider. SES, Twilio, FCM all support idempotency tokens. (b) For internal retries within a decorator, the underlying send is what's idempotent; the decorator's retries reuse the same notification including its key.
  • Sync vs async send? Sync is the example. Production systems queue notifications and process them asynchronously. The Channel interface can stay sync; an AsyncChannel decorator queues and returns immediately. Or the whole system shifts to send returning a CompletableFuture. The latter composes better with other async code; the former is simpler.
  • How do you test all this composition? Each decorator is independently testable. Test RetryingChannel with a fake Channel that fails N-1 times; assert it retries and eventually succeeds. Test LoggingChannel with a fake; assert it logs. The composition is just plumbing — its tests are smoke tests at integration level.

№ 03

Hotel booking

~45 min Domain Medium
Prompt

Model a hotel booking system. A hotel has rooms of different types (standard, deluxe, suite). A guest can hold a reservation for some date range, then confirm it (paid) or let the hold expire. Reservations can be cancelled before they're completed. Two reservations on the same room can never have overlapping date ranges.

Walk through: what are the entities, what state do they hold, where do the rules live, how does the system enforce the no-overlap invariant under concurrent bookings.

Clarify
  • What does "hold" mean exactly — soft hold (some other guest can override) or hard hold (TTL-based)?
  • Are reservations tied to specific room IDs at hold time, or just to a room type?
  • Are partial-day bookings supported, or always integer days?
  • Is overbooking ever allowed (some hotels intentionally allow this)?
Approach
▸ Design thinking

This problem is testing your domain modeling instincts. The actual code is short; the choice of which entities exist and what state they hold is the entire game.

Start by listing nouns: hotel, room, room-type, guest, reservation, date-range. Then verbs: hold, confirm, cancel, expire, complete. Then invariants: no two non-cancelled reservations on the same room can overlap in time.

The lifecycle decision: a reservation has multiple states. The naive "boolean flags" approach (isHeld, isConfirmed, isCancelled) creates impossible states (held AND cancelled?) and is unparseable as it grows. A single Status enum makes the states exhaustive and exclusive — there is exactly one state at a time, and the type system enforces it. Small win that pays back constantly.

Where the rules live: not on the entities. The Reservation record is data — id, dates, status, hold timeout. The BookingService is where rules are enforced: "you can only confirm a HELD reservation," "no overlap on this room for active reservations." This separation is sometimes called the "anemic domain model" anti-pattern, but for booking-style systems where rules involve multiple entities (find a free room, check it doesn't overlap), the service-centric design is cleaner than trying to put cross-entity rules on a single entity.

Repository as a boundary: RoomRepository and ReservationRepository are interfaces. The service depends on them, not on the database. Tests pass in-memory implementations; production passes JDBC/JPA implementations. The service code is identical. This is dependency inversion in concrete form — the service points "down" at abstractions; the database adapter points "up" at the same abstractions.

The concurrency issue (worth raising even though this isn't a pure-LLD problem): two guests booking the last room simultaneously. Both conflicts() checks return false; both save()s succeed; you've double-booked. Two real fixes: (a) database-level — unique constraint on (roomId, dateRange) using PostgreSQL's exclusion constraints, or row-level locking. (b) application-level — pessimistic lock per room. Mention both and explain why the database fix is preferred (correctness without trusting application code).

Solution
// Hotel booking: domain modeling.
import java.time.Instant;
import java.time.LocalDate;
import java.util.*;

public final class Booking {

    public record DateRange(LocalDate checkIn, LocalDate checkOut) {
        public boolean overlaps(DateRange o) {
            return checkIn.isBefore(o.checkOut) && o.checkIn.isBefore(checkOut);
        }
    }

    public enum RoomType { STANDARD, DELUXE, SUITE }
    public record Room(String id, RoomType type) {}

    public enum Status { HELD, CONFIRMED, EXPIRED, CANCELLED, COMPLETED }

    public record Reservation(String id, String roomId, String guestId,
                              DateRange range, Status status, Instant heldUntil) {}

    public static class BookingException extends RuntimeException {
        public BookingException(String m) { super(m); }
    }

    // Repository = the persistence boundary. The booking service doesn't know
    // or care if rooms live in memory or in a database.
    public interface RoomRepository {
        List<Room> findByType(RoomType type);
    }

    public interface ReservationRepository {
        List<Reservation> activeForRoom(String roomId);
        void save(Reservation r);
        Optional<Reservation> find(String id);
    }

    // The service coordinates between repositories and enforces invariants.
    public static final class BookingService {
        private final RoomRepository rooms;
        private final ReservationRepository reservations;
        private long idCounter = 0;

        public BookingService(RoomRepository r, ReservationRepository res) {
            this.rooms = r; this.reservations = res;
        }

        public Reservation hold(String guestId, RoomType type, DateRange range,
                                long holdSeconds) {
            for (Room room : rooms.findByType(type)) {
                if (!conflicts(room.id(), range)) {
                    Reservation r = new Reservation(
                        "R" + (++idCounter), room.id(), guestId, range,
                        Status.HELD, Instant.now().plusSeconds(holdSeconds));
                    reservations.save(r);
                    return r;
                }
            }
            throw new BookingException("no room of type available");
        }

        public Reservation confirm(String reservationId) {
            Reservation r = reservations.find(reservationId)
                .orElseThrow(() -> new BookingException("not found"));
            if (r.status() != Status.HELD)
                throw new BookingException("not in HELD state");
            if (r.heldUntil().isBefore(Instant.now()))
                throw new BookingException("hold expired");
            Reservation confirmed = new Reservation(r.id(), r.roomId(), r.guestId(),
                r.range(), Status.CONFIRMED, r.heldUntil());
            reservations.save(confirmed);
            return confirmed;
        }

        private boolean conflicts(String roomId, DateRange range) {
            Instant now = Instant.now();
            return reservations.activeForRoom(roomId).stream()
                .filter(r -> r.status() == Status.CONFIRMED ||
                           (r.status() == Status.HELD && r.heldUntil().isAfter(now)))
                .anyMatch(r -> r.range().overlaps(range));
        }
    }
}

// Reservation lifecycle:
//   HELD (reserved temporarily) -> CONFIRMED (paid) -> COMPLETED (after stay)
//   HELD -> EXPIRED (timeout)
//   any -> CANCELLED
enum class Status { HELD, CONFIRMED, EXPIRED, CANCELLED, COMPLETED };

struct Reservation {
    std::string id;
    std::string roomId;
    std::string guestId;
    DateRange range;
    Status status;
    std::chrono::steady_clock::time_point heldUntil;  // for HELD only
};

class BookingException : public std::runtime_error {
public:
    using std::runtime_error::runtime_error;
};

// Repository = the persistence boundary. The booking service doesn't know
// or care if rooms live in memory or in a database.
class RoomRepository {
public:
    virtual ~RoomRepository() = default;
    virtual std::vector<Room> findByType(RoomType type) const = 0;
};

class ReservationRepository {
public:
    virtual ~ReservationRepository() = default;
    virtual std::vector<Reservation> activeForRoom(const std::string& roomId) const = 0;
    virtual void save(const Reservation& r) = 0;
    virtual std::optional<Reservation> find(const std::string& id) const = 0;
};

// In-memory implementations (for the example/tests)
class InMemRoomRepo : public RoomRepository {
    std::vector<Room> rooms_;
public:
    void add(Room r) { rooms_.push_back(std::move(r)); }
    std::vector<Room> findByType(RoomType type) const override {
        std::vector<Room> out;
        for (auto& r : rooms_) if (r.type == type) out.push_back(r);
        return out;
    }
};

class InMemResRepo : public ReservationRepository {
    std::unordered_map<std::string, Reservation> map_;
public:
    std::vector<Reservation> activeForRoom(const std::string& roomId) const override {
        std::vector<Reservation> out;
        auto now = std::chrono::steady_clock::now();
        for (auto& [_, r] : map_) {
            if (r.roomId != roomId) continue;
            if (r.status == Status::CONFIRMED) out.push_back(r);
            if (r.status == Status::HELD && r.heldUntil > now) out.push_back(r);
        }
        return out;
    }
    void save(const Reservation& r) override { map_[r.id] = r; }
    std::optional<Reservation> find(const std::string& id) const override {
        auto it = map_.find(id);
        if (it == map_.end()) return std::nullopt;
        return it->second;
    }
};

// The service coordinates between repositories and enforces invariants.
// All business logic lives here, not in the entities.
class BookingService {
    RoomRepository& rooms_;
    ReservationRepository& reservations_;
    int idCounter_ = 0;

    bool conflicts(const std::string& roomId, DateRange range) {
        for (auto& r : reservations_.activeForRoom(roomId)) {
            if (r.range.overlaps(range)) return true;
        }
        return false;
    }
public:
    BookingService(RoomRepository& r, ReservationRepository& res)
        : rooms_(r), reservations_(res) {}

    Reservation hold(const std::string& guestId, RoomType type, DateRange range,
                     std::chrono::seconds holdFor) {
        for (auto& room : rooms_.findByType(type)) {
            if (!conflicts(room.id, range)) {
                Reservation r;
                r.id = "R" + std::to_string(++idCounter_);
                r.roomId = room.id;
                r.guestId = guestId;
                r.range = range;
                r.status = Status::HELD;
                r.heldUntil = std::chrono::steady_clock::now() + holdFor;
                reservations_.save(r);
                return r;
            }
        }
        throw BookingException("no room of type available for the dates");
    }

    Reservation confirm(const std::string& reservationId) {
        auto r = reservations_.find(reservationId);
        if (!r) throw BookingException("not found");
        if (r->status != Status::HELD) throw BookingException("not in HELD state");
        if (r->heldUntil < std::chrono::steady_clock::now())
            throw BookingException("hold expired");
        r->status = Status::CONFIRMED;
        reservations_.save(*r);
        return *r;
    }
};

int main() {
    InMemRoomRepo rooms;
    rooms.add({"101", RoomType::STANDARD});
    rooms.add({"102", RoomType::STANDARD});
    InMemResRepo res;
    BookingService svc(rooms, res);

    auto r1 = svc.hold("guest-A", RoomType::STANDARD, {1, 4},
                       std::chrono::seconds(60));
    std::cout << "held " << r1.id << " on room " << r1.roomId << "\n";

    auto confirmed = svc.confirm(r1.id);
    std::cout << "confirmed " << confirmed.id << "\n";

    try {
        svc.hold("guest-B", RoomType::STANDARD, {2, 5}, std::chrono::seconds(60));
    } catch (const BookingException& e) {
        std::cout << "expected conflict: " << e.what() << "\n";
    }

    // A non-overlapping booking still succeeds (other room or other dates)
    auto r3 = svc.hold("guest-C", RoomType::STANDARD, {5, 7},
                       std::chrono::seconds(60));
    std::cout << "held " << r3.id << " on room " << r3.roomId << "\n";
    return 0;
}
▸ Discussion / Follow-ups (with answers)
  • How do you prevent double-booking under concurrency? Application-level locking (per-room mutex) works on a single instance but fails across instances. The right answer is a database constraint. PostgreSQL's exclusion constraints can directly express "no two rows for the same room have overlapping date ranges" — the database refuses to commit conflicting rows. Without that, you need pessimistic locking on the room row inside a transaction. Don't trust application code for this invariant; let the database enforce it.
  • How do held reservations expire? Two approaches. (a) Lazy: activeForRoom filters out expired holds at query time. The example does this. Simple, no background process, but DB rows accumulate. (b) Active: a scheduled job sweeps expired holds and updates them to EXPIRED. Cleaner data; needs a job runner. Most systems do both — lazy for correctness, active for cleanup.
  • Why an enum for status, not separate boolean flags? Because the states are exclusive. (isHeld=true, isCancelled=true) is a nonsense state; the boolean encoding lets you represent it. The enum makes invalid states unrepresentable. This is the small-but-important "make illegal states unrepresentable" rule from type-driven design.
  • Where do business rules belong — entity or service? Depends on whether the rule needs other entities. "Reservation can only be confirmed if HELD" — entity rule (or service; both work). "No overlap on this room" — service rule, because it needs to query other reservations. Don't try to put cross-entity rules on a single entity; you'll end up passing repositories into entity methods, which inverts the dependency badly.
  • How does the service stay testable when the database is involved? The repositories are interfaces. Tests pass in-memory implementations. Service-level tests run in milliseconds. Integration tests against the real database verify the repository implementations and the database constraints. This is the test pyramid as it shows up in domain-driven design.
  • What about partial-day bookings, time zones, daylight saving? If the prompt says "date range," use LocalDate (no time, no timezone). If it's check-in/check-out times, use ZonedDateTime with the hotel's timezone — guests' timezones are irrelevant; the hotel's clock is what matters. DST is its own headache; a 24-hour stay across a DST boundary is 23 or 25 hours.
  • How would you support overbooking? Some hotels intentionally allow it (statistical no-shows). The conflicts check becomes "is the count of overlapping reservations >= overbookingThreshold?" instead of "> 0". Per-room-type configuration. The structure stays the same.

№ 04

Logging library

~40 min Composition Medium
Prompt

Build the core of a structured logging library. It needs: log levels (TRACE through ERROR), multiple output destinations (stderr, files, network), per-destination formatting (plain text, JSON, structured key-value), and per-destination filters (only emit WARN+ to stderr, only emit specific loggers to a file).

The interviewer is looking for: how do you keep formatters, sinks, and filters orthogonal? Can a sink have its own formatter? Can a logger fan out to multiple sinks with different filters?

Clarify
  • Are loggers hierarchical (com.foo vs com.foo.bar) or flat?
  • Is async emission required, or is synchronous fine?
  • Should formatters be reusable across sinks, or coupled?
  • Do filters compose (an "ALL" of multiple filters)?
Approach
▸ Design thinking

This is composition done right — each piece does one thing, and they slot together.

Three orthogonal concepts:

  • Formatter: turns a LogRecord into a string. Plain, JSON, key-value. Pure function.
  • Filter: decides if a record should be emitted. Level filter, logger-name filter, MDC filter. Boolean function.
  • Sink: the actual output destination. Stderr, file, network. Has-a Formatter, has-a list of Filters.

The Logger fans out: a single record gets dispatched to multiple sinks. Each sink applies its own filters and formatter independently. So a setup like "stderr at WARN+ in plain text, JSON file at INFO+, network sink at ERROR+ in JSON" is just three sink instances configured differently — no special-casing in the logger.

Why this design beats a monolithic Logger: each axis varies independently. Want to swap how warnings are formatted? New formatter, plug into the warnings sink. Want to add a Datadog sink? New sink class. Want to filter out a noisy logger? Add a filter. None of these changes touch the others.

The trap the interviewer is looking for you to avoid: making the Logger know about formats. logger.logJson(...) versus logger.logPlain(...) bakes the format into the call site. The right design has the call site say what happened; sinks decide how to render it.

Async emission is a fair follow-up. The synchronous design here blocks the calling thread on I/O. Production loggers (log4j2, logback) use async appenders with bounded queues. The structure doesn't change — just add an AsyncSink decorator that wraps a sink and dispatches via an internal worker thread.

Solution
// Logging: composition (sinks, formatters, filters).
import java.io.PrintStream;
import java.time.Instant;
import java.util.ArrayList;
import java.util.List;

public final class Logging {

    public enum Level { TRACE, DEBUG, INFO, WARN, ERROR }

    public record LogRecord(Level level, String message, String loggerName, Instant ts) {}

    // Different sinks may want different formats.
    public interface Formatter { String format(LogRecord r); }

    public static final class PlainFormatter implements Formatter {
        public String format(LogRecord r) {
            return "[" + r.level() + "] " + r.loggerName() + " - " + r.message();
        }
    }

    public static final class JsonFormatter implements Formatter {
        public String format(LogRecord r) {
            return String.format(
                "{\"level\":\"%s\",\"logger\":\"%s\",\"msg\":\"%s\"}",
                r.level(), r.loggerName(), r.message());
        }
    }

    public interface Filter { boolean accepts(LogRecord r); }

    public static final class LevelFilter implements Filter {
        private final Level min;
        public LevelFilter(Level min) { this.min = min; }
        public boolean accepts(LogRecord r) {
            return r.level().ordinal() >= min.ordinal();
        }
    }

    // Sink owns its formatter and filters. Multiple sinks can co-exist.
    public interface Sink { void emit(LogRecord r); }

    public static final class StreamSink implements Sink {
        private final PrintStream out;
        private final Formatter fmt;
        private final List<Filter> filters = new ArrayList<>();
        public StreamSink(PrintStream out, Formatter fmt) {
            this.out = out; this.fmt = fmt;
        }
        public void addFilter(Filter f) { filters.add(f); }
        public void emit(LogRecord r) {
            for (Filter f : filters) if (!f.accepts(r)) return;
            synchronized (this) { out.println(fmt.format(r)); }
        }
    }

    // Logger fans out to all sinks. Each sink applies its own filters.
    public static final class Logger {
        private final String name;
        private final List<Sink> sinks = new ArrayList<>();
        public Logger(String name) { this.name = name; }
        public void addSink(Sink s) { sinks.add(s); }
        public void log(Level l, String msg) {
            LogRecord r = new LogRecord(l, msg, name, Instant.now());
            for (Sink s : sinks) s.emit(r);
        }
        public void info(String m)  { log(Level.INFO, m); }
        public void warn(String m)  { log(Level.WARN, m); }
        public void error(String m) { log(Level.ERROR, m); }
    }
}

public:

class PlainFormatter : public Formatter {
public:

        os << "[" << levelName(r.level) << "] " << r.loggerName << " - " << r.message;
        return os.str();
    }
};

class JsonFormatter : public Formatter {
public:
    std::string format(const LogRecord& r) const override {
        std::ostringstream os;
        os << "{\"level\":\"" << levelName(r.level)
           << "\",\"logger\":\"" << r.loggerName
           << "\",\"msg\":\"" << r.message << "\"}";
        return os.str();
    }
};

// A Filter decides whether a record should be emitted. Composes via a chain.
class Filter {
public:
    virtual ~Filter() = default;
    virtual bool accepts(const LogRecord& r) const = 0;
};

class LevelFilter : public Filter {
    Level min_;
public:
    explicit LevelFilter(Level min) : min_(min) {}
    bool accepts(const LogRecord& r) const override {
        return static_cast<int>(r.level) >= static_cast<int>(min_);
    }
};

// A Sink is the final destination — stderr, file, network. A sink owns
// its formatter and filters; this lets you have a stderr sink at WARN+
// running in plain format alongside a file sink at INFO+ in JSON, etc.
class Sink {
public:
    virtual ~Sink() = default;
    virtual void emit(const LogRecord& r) = 0;
};

class StreamSink : public Sink {
    std::ostream& out_;
    std::shared_ptr<Formatter> fmt_;
    std::vector<std::shared_ptr<Filter>> filters_;
    std::mutex mu_;
public:
    StreamSink(std::ostream& out, std::shared_ptr<Formatter> fmt)
        : out_(out), fmt_(std::move(fmt)) {}
    void addFilter(std::shared_ptr<Filter> f) { filters_.push_back(std::move(f)); }
    void emit(const LogRecord& r) override {
        for (auto& f : filters_) if (!f->accepts(r)) return;
        std::lock_guard<std::mutex> lock(mu_);
        out_ << fmt_->format(r) << "\n";
    }
};

// The Logger is the entry point. A logger has a name and a list of sinks;
// the same record fans out to every sink. Each sink applies its own filters.
class Logger {
    std::string name_;
    std::vector<std::shared_ptr<Sink>> sinks_;
public:
    explicit Logger(std::string name) : name_(std::move(name)) {}
    void addSink(std::shared_ptr<Sink> s) { sinks_.push_back(std::move(s)); }
    void log(Level l, const std::string& msg) {
        LogRecord r{l, msg, name_, std::chrono::system_clock::now()};
        for (auto& s : sinks_) s->emit(r);
    }
    void info(const std::string& m)  { log(Level::INFO, m); }
    void warn(const std::string& m)  { log(Level::WARN, m); }
    void error(const std::string& m) { log(Level::ERROR, m); }
};

    stderrSink->addFilter(std::make_shared<LevelFilter>(Level::WARN));

    auto stdoutSink = std::make_shared<StreamSink>(std::cout, std::make_shared<JsonFormatter>());
    stdoutSink->addFilter(std::make_shared<LevelFilter>(Level::INFO));

    Logger log("my-service");
    log.addSink(stderrSink);
    log.addSink(stdoutSink);

    log.info("starting up");        // -> stdout (JSON)
    log.warn("something off");      // -> stderr + stdout
    log.error("shit happened");     // -> stderr + stdout
    log.log(Level::DEBUG, "dbg");   // -> nowhere (filtered)
    return 0;
}
▸ Discussion / Follow-ups (with answers)
  • Why isn't this just System.out.println? Because production logging needs filtering by level (don't ship DEBUG to stderr in prod), structured output for log aggregators (JSON), multiple destinations (stderr + file + Datadog), and per-environment config (verbose in dev, quiet in prod). A printf can't do any of those.
  • How do you make this async without changing the API? Wrap a sink with an AsyncSink decorator. The async sink owns a bounded queue and a worker thread; emit just enqueues. The StreamSink doesn't change. This is the async appender pattern from log4j2. Trade-off: events can be lost if the queue fills (drop policy) or block writers (block policy) — pick one.
  • How do you handle MDC (mapped diagnostic context) — request IDs, user IDs? Thread-local context. The logger reads from MDC when constructing the LogRecord; formatters can render it. This lets you correlate logs across a request without threading the request ID through every method call. SLF4J/Logback do this; the design extends naturally.
  • What about hierarchical loggers (com.foo, com.foo.bar)? A logger registry organized as a tree. Looking up com.foo.bar walks up the tree to find the most-specific configuration; missing levels inherit. This is what log4j and Logback do. Adds a level of indirection; useful for fine-grained per-package config.
  • How do you avoid the cost of log message formatting when the level is filtered out? Lazy formatting. Pass a supplier (() -> "expensive: " + computeStuff()) instead of an eager string. The logger only invokes the supplier if the level passes the filter. Java SLF4J's logger.info("user {} did {}", user, action) uses string templates that defer interpolation. C++ libraries (spdlog, fmt-based loggers) do the same with format strings.
  • Why fan-out to multiple sinks rather than one configurable sink? Because the destinations have different lifecycles, different config, different formats. A "configurable sink" that handles all of these ends up reimplementing fan-out internally. Make it explicit: one logger, multiple sinks; each sink stands alone.
  • How do you reload config (e.g., bump verbosity at runtime)? The Logger holds references to sinks; sinks hold references to filters and formatters. Replace the filters/formatters via setters under a lock; the logger picks up the new config on the next call. log4j2 does live reloading via file watchers; the structure here supports it without modification.

Prompt

You inherit a 800-line OrderProcessor class. Approach
▸ Design thinking

The diagnosis comes first; the cure follows. List specifically what's wrong:

  1. Single Responsibility violation. OrderProcessor is at least seven different services smashed together. Inventory, fraud, payment, persistence, email, analytics, audit — each is a real concern that other code in the system also needs. By coupling them in one class, every change risks breaking unrelated callers.
  2. Hidden dependencies. The 800-line method probably constructs collaborators inline (new EmailService(), new PaymentGateway(config.get("..."))). This is why mocking is hell — you can't substitute fakes without modifying the class itself. Test setup is a maze of static mocks and reflection tricks.
  3. Test smell as the leading indicator. "Tests are slow and flaky" is the telltale. Slow tests come from real I/O happening because dependencies aren't mockable. Flaky tests come from timing or state shared between tests. Both are symptoms of the dependencies being constructed by the unit under test instead of injected.

The refactor in two moves:

Move 1: extract collaborators behind interfaces. InventoryService, PaymentGateway, OrderRepository, NotificationService — each becomes an interface plus an implementation. The OrderProcessor takes them via constructor. This is dependency injection at its most basic: you don't construct what you depend on. Tests pass fakes; production passes real ones.

Move 2: turn the orchestrator into pure coordination. The new process() method is now ~30 lines: call inventory.reserve, call payment.charge, save order, notify. No I/O directly; no business logic; just sequencing and compensation (rollback on failure). The compensating logic — release inventory if payment fails, refund payment if save fails — is the only thing that must live in the orchestrator, because only the orchestrator knows the cross-collaborator order of operations.

What stays a single class: the orchestration. Don't fragment that into a half-dozen tiny classes; you'll have spread the orchestration logic across files and made it harder to reason about. The collaborators get their own classes; the coordinator stays one class.

What this buys you: tests run in milliseconds with in-memory fakes. Adding a new step (fraud check) is adding a constructor parameter and one line in process. Two engineers can change different collaborators without conflicting.

Approach
▸ Design thinking

The principle: code at the application level should depend on capabilities, not on backends. The reporting code wants to "save bytes under a key and read them back later." Whether those bytes live on disk, in S3, or in memory is irrelevant to the reporting logic. So: define an interface that captures the capability, then have backends implement it.

The abstraction is small but worth getting right:

  • put(key, bytes), get(key) -> Optional<bytes>, remove(key), list(prefix).
  • Returns Optional for "not found" rather than throwing — missing keys aren't exceptional, they're a normal outcome.
  • list takes a prefix because that's the common case (S3, hierarchical filesystems both support it efficiently).

Then write the in-memory backend FIRST. This is a load-bearing tip. If the abstraction is well-defined, you can implement it with an in-memory Map in 20 lines, and your tests can run against it. If you find yourself unable to implement it without referring to S3 specifics, your abstraction has leaked. Fix the abstraction, not the backend.

The hard question the interviewer is fishing for: do you expose backend-specific capabilities, or stay lowest-common-denominator? Both have failure modes:

  • Lowest-common-denominator: callers can't take advantage of S3 multipart uploads, can't set object metadata, can't generate presigned URLs. They write efficient code for the worst-case backend.
  • Expose everything: the interface bloats; the in-memory backend has to fake or stub features it doesn't really support; portability dies.

The pragmatic answer is capability flags or extended interfaces: FileSystem for the common 80%; SupportsMultipart, SupportsPresignedUrls as opt-in interfaces backends may also implement. Callers check via instanceof when they need the capability. This is closer to how AWS SDKs and storage libraries actually shape things.

Feature flag system

~45 min Extension Medium
Prompt

This problem has four parts. Build each, then watch the design either survive or break:

  1. v1: a flag service where each flag is a boolean that's either on or off globally. isEnabled("feature.x").
  2. v2: percentage rollouts. "Enable for 10% of users." Same user always gets the same answer.
  3. v3: explicit user targeting. "Always on for these specific users."
  4. v4: composition. "On for these specific users OR for 25% of everyone else."

If your v1 needs rewriting at v2, your design didn't anticipate growth. If your v3 forces v1 callers to change, you broke them. If v4 requires touching v1/v2/v3, you missed the abstraction.

Clarify
  • Is the percentage assignment stable per user (always same answer) or per-call random?
  • Are flag evaluations expected to be cheap (microseconds) or is a network call OK?
  • Will flags be loaded from JSON config, a database, or a remote service?
  • Do we need flag-level audit (who changed what, when)?
Approach
▸ Design thinking

This problem is about designing for change. The trap is solving v1 in a way that survives until v2 but rewrites at v3. Each version of "now add this" is a real production pattern someone has hit; getting them right up front is what mid-level vs senior is testing.

The key move at v1: don't write boolean isEnabled(String flag) with a Map of strings to booleans. Even though that's all v1 needs. Instead, model a flag as an object that knows how to evaluate itself:

interface FlagRule {
    boolean evaluate(UserContext ctx);
}

For v1, the implementation is BooleanFlag(true) or BooleanFlag(false). It evaluates the same way regardless of context. But the API has the UserContext in it from day one, even though v1 doesn't need it.

This is the only design decision that matters. Now:

  • v2 (percentage): write PercentageFlag(flagId, pct) that hashes the user ID and compares to threshold. The service code is unchanged; the registry just contains a different FlagRule.
  • v3 (user targeting): write UserTargetFlag(allowedUserIds). Same deal.
  • v4 (composition): write AnyFlag(rules) that ORs its children. Now you can register a flag that's the OR of UserTargetFlag and PercentageFlag. Existing flags untouched.

The pattern at the heart of this is the Strategy pattern with a recursive variant (since composites are themselves rules). It's the same shape as the rule engine from the previous practice set, the same shape as the price modifier pipeline above. Recognizing this shape across problems is the senior signal.

What if you'd done it the boolean-Map way? v2 forces a rewrite — the Map values are now functions, all callers see API changes. v3 forces another rewrite. v4 is impossible without writing v1+v2+v3 a third time. You can do this work, but you do it three times, and every step risks regressing the previous step's behavior in production.

Stable bucketing deserves a callout. Percentage rollouts must be deterministic per user — a user assigned to the 25%-on bucket today must still be there tomorrow, or you'll thrash. Hashing (flagId + ":" + userId) gives a stable bucket. flagId is in the hash so different flags assign the same user differently — otherwise the same 25% of users gets every "25% rollout" feature, which biases your metrics.

Solution

import java.util.*;

public final class Flags {

    public record UserContext(String userId, String country,
                              Map<String, String> attributes) {}

    // A flag's evaluation is a function from context to bool. Polymorphic
    // interface (rather than switching on flag-type) means new flag kinds
    // are add-only — no existing code changes.
    public interface FlagRule {
        boolean evaluate(UserContext ctx);
    }

    public static final class BooleanFlag implements FlagRule {
        private final boolean value;
        public BooleanFlag(boolean v) { this.value = v; }
        public boolean evaluate(UserContext ctx) { return value; }
    }

    // Stable per-user, deterministic. flagId in the hash so different flags
    // assign the same user differently — otherwise the same 25% gets every
    // 25% rollout and your metrics are biased.
    public static final class PercentageFlag implements FlagRule {
        private final String flagId;
        private final int percentage;
        public PercentageFlag(String id, int pct) { this.flagId = id; this.percentage = pct; }
        public boolean evaluate(UserContext ctx) {
            int bucket = Math.floorMod(
                (flagId + ":" + ctx.userId()).hashCode(), 100);
            return bucket < percentage;
        }
    }

    public static final class UserTargetFlag implements FlagRule {
        private final Set<String> allowed;
        public UserTargetFlag(Set<String> allowed) { this.allowed = Set.copyOf(allowed); }
        public boolean evaluate(UserContext ctx) { return allowed.contains(ctx.userId()); }
    }

    // ANY of the inner rules. Combines targeting + percentage cleanly.
    public static final class AnyFlag implements FlagRule {
        private final List<FlagRule> rules;
        public AnyFlag(List<FlagRule> rules) { this.rules = List.copyOf(rules); }
        public boolean evaluate(UserContext ctx) {
            return rules.stream().anyMatch(r -> r.evaluate(ctx));
        }
    }

    public static final class FlagService {
        private final Map<String, FlagRule> flags = new HashMap<>();
        public void register(String id, FlagRule rule) { flags.put(id, rule); }
        public boolean isEnabled(String flagId, UserContext ctx) {
            FlagRule r = flags.get(flagId);
            return r != null && r.evaluate(ctx);     // safe default: false
        }
    }
}

Order workflow

~55 min Domain Hard
Prompt

An order moves through states: DRAFT → SUBMITTED → PAID → FULFILLING → SHIPPED → DELIVERED. It can also be CANCELLED (from several states) or RETURNED (only after delivery). Each transition has rules: payment requires a non-zero total, shipping requires a confirmed address, returning requires being within the return window. The product team will keep adding states and rules.

The interviewer wants to see: where do the rules live, how do invalid transitions get prevented, how do downstream systems (audit logs, notifications, inventory) hear about transitions, and how do you add a new state next quarter without rewriting anything?

Clarify
  • Are state transitions transactional with side effects (inventory decrement happens IFF state advances)?
  • Can multiple transitions happen concurrently on the same order?
  • Should the state machine be configurable per order type, or is it global?
  • How do downstream systems learn about transitions — events, polling, callbacks?
Approach
▸ Design thinking

You can model an order's lifecycle three ways. Two are wrong; one is right.

Wrong way #1 — boolean flags: isPaid, isShipped, isCancelled. The state is a tuple of booleans. Impossible states (paid AND cancelled?) are representable. New states require touching every code path. This is what most undisciplined codebases do; it rots fast.

Wrong way #2 — single string field, conditionals everywhere: order.status = "shipped". The transitions live in scattered if statements: "if order.status == 'paid', advance to shipped." There's no central place to ask "what transitions are allowed from this state?" Reading the code, you can't tell.

Right way — explicit state machine: states are an enum, transitions are data (a map from (state, event) to next state), the engine is generic. The product team's "add a new state" becomes one line of configuration; the engine is unchanged. Invalid transitions are detected by lookup miss, not by scattered if guards.

Three pieces:

  1. The states: an enum. Closed set; compiler-enforced exhaustiveness.
  2. The transitions: a map. Configurable per order type if needed.
  3. The engine: generic. fire(order, event) looks up (order.state, event), throws if missing, advances if present, fires listeners.

Side effects on transitions are the hardest follow-up. When PAID → FULFILLING happens, you need to: decrement inventory, dispatch a fulfillment job, write to the audit log. Two design choices:

  • Listeners on the state machine — the design here. Every transition fires a listener; subscribers handle their concerns. Decoupled, easy to add new listeners. Trade-off: side effects aren't transactional with the state change. If the audit log fails after inventory decrements, you have inconsistency.
  • Domain events — the state change emits an event ("OrderPaid"); a transactional outbox writes the event to a queue in the same DB transaction as the state change. Consumers read from the queue. Atomic with the state change; consumers are decoupled. This is the right answer in production but more complex.

For an interview, the listener design is correct and you mention the outbox pattern as the production hardening.

Concurrency: two events firing on the same order simultaneously can cause torn states. Solutions: optimistic locking with a version column in the database; the state machine takes (order, version, event) and CASes the version atomically with the state update.

Solution

import java.util.*;

public final class Workflow {

    public enum OrderState {
        DRAFT, SUBMITTED, PAID, FULFILLING, SHIPPED, DELIVERED, CANCELLED, RETURNED
    }

    public static final class Order {
        public final String id;
        public OrderState state = OrderState.DRAFT;
        public double total;
        public Order(String id, double total) { this.id = id; this.total = total; }
    }

    public static class IllegalTransition extends RuntimeException {
        public IllegalTransition(String m) { super(m); }
    }

    // The state machine is data — a map from (state, event) to next-state.
    // Adding a transition is a one-line config change; engine unchanged.
    public static final class OrderStateMachine {
        private record Key(OrderState from, String event) {}

        private final Map<Key, OrderState> transitions = new HashMap<>();
        private final List<TransitionListener> listeners = new ArrayList<>();

        @FunctionalInterface
        public interface TransitionListener {
            void onTransition(Order o, OrderState from, OrderState to);
        }

        public OrderStateMachine allow(OrderState from, String event, OrderState to) {
            transitions.put(new Key(from, event), to);
            return this;
        }

        public void onTransition(TransitionListener l) { listeners.add(l); }

        public void fire(Order o, String event) {
            OrderState next = transitions.get(new Key(o.state, event));
            if (next == null) {
                throw new IllegalTransition(
                    "no transition from " + o.state + " on '" + event + "'");
            }
            OrderState before = o.state;
            o.state = next;
            for (TransitionListener l : listeners) l.onTransition(o, before, next);
        }
    }

    public static OrderStateMachine buildOrderMachine() {
        return new OrderStateMachine()
            .allow(OrderState.DRAFT, "submit", OrderState.SUBMITTED)
            .allow(OrderState.SUBMITTED, "pay", OrderState.PAID)
            .allow(OrderState.PAID, "start_fulfilling", OrderState.FULFILLING)
            .allow(OrderState.FULFILLING, "ship", OrderState.SHIPPED)
            .allow(OrderState.SHIPPED, "deliver", OrderState.DELIVERED)
            .allow(OrderState.DELIVERED, "return", OrderState.RETURNED)

            .allow(OrderState.DRAFT, "cancel", OrderState.CANCELLED)
            .allow(OrderState.SUBMITTED, "cancel", OrderState.CANCELLED)
            .allow(OrderState.PAID, "cancel", OrderState.CANCELLED);

    double total = 0;

class IllegalTransition : public std::runtime_error {
public:
    using std::runtime_error::runtime_error;

// The state machine itself is a piece of data: a map from (state, event)
// to next-state. This is the engine; the rules are configuration.
class OrderStateMachine {
public:
    using Event = std::string;

private:
    struct Key {
        OrderState from; Event ev;
        bool operator==(const Key& o) const { return from == o.from && ev == o.ev; }
    };
    struct KeyHash {
        size_t operator()(const Key& k) const {
            return std::hash<int>()(static_cast<int>(k.from)) ^ std::hash<std::string>()(k.ev);
        }
    };

    std::unordered_map<Key, OrderState, KeyHash> transitions_;
    std::vector<std::function<void(const Order&, OrderState, OrderState)>> listeners_;

public:
    OrderStateMachine& allow(OrderState from, const Event& ev, OrderState to) {
        transitions_[{from, ev}] = to;
        return *this;
    }

    void onTransition(std::function<void(const Order&, OrderState, OrderState)> f) {
        listeners_.push_back(std::move(f));
    }

    void fire(Order& o, const Event& ev) {
        auto it = transitions_.find({o.state, ev});
        if (it == transitions_.end()) {
            throw IllegalTransition(
                "no transition from " + stateName(o.state) + " on '" + ev + "'");
        }
        OrderState before = o.state;
        o.state = it->second;
        for (auto& l : listeners_) l(o, before, o.state);
    }
};

    OrderStateMachine m;
    m.allow(OrderState::DRAFT, "submit", OrderState::SUBMITTED)
     .allow(OrderState::SUBMITTED, "pay", OrderState::PAID)
     .allow(OrderState::PAID, "start_fulfilling", OrderState::FULFILLING)
     .allow(OrderState::FULFILLING, "ship", OrderState::SHIPPED)
     .allow(OrderState::SHIPPED, "deliver", OrderState::DELIVERED)
     .allow(OrderState::DELIVERED, "return", OrderState::RETURNED)
     // Cancellations from multiple states
     .allow(OrderState::DRAFT, "cancel", OrderState::CANCELLED)
     .allow(OrderState::SUBMITTED, "cancel", OrderState::CANCELLED)
     .allow(OrderState::PAID, "cancel", OrderState::CANCELLED);
    return m;
}

    });

  • Why a state machine over a workflow engine (BPMN, Camunda, Temporal)? Scale of the problem. Simple state machines fit in-process and are easy to reason about. Workflow engines are right when you have long-running business processes (days to months), distributed coordination, or business analysts modeling the flow themselves. For an order's lifecycle, in-process is usually the right answer; for a multi-step approval workflow with humans in the loop, an engine is the right answer.
  • How do you handle side effects atomically with the state change? Transactional outbox pattern. The state change and an event row are written in the same database transaction. A separate process reads the event row and publishes to a message bus. Consumers (inventory, audit, notifications) react to events. Atomicity between state and event-publication; consumers are decoupled. This is what real-world e-commerce systems do.
  • How do you deal with concurrent transitions on the same order? Optimistic locking. The order has a version column. Transitions read the order with version, fire, save with version + 1; the database rejects the save if the version has changed. The losing transaction retries or reports conflict. Pessimistic locking (SELECT FOR UPDATE) also works; pick based on contention level.
  • How do you support "undo" of a transition? Carefully. Transitions usually have side effects; "undoing" requires compensating effects. Two patterns: (a) Wide entity — Order has all fields; ones not relevant to the current state are null. Simple; tolerable for small numbers of fields. (b) State-specific data — separate tables (order_shipping_info) related to the order, populated when relevant. Cleaner but more joins. Pick based on field count.

Pluggable serialization

~40 min Trade-Offs Medium
Prompt

Your service needs to serialize domain objects (orders, products, users) to JSON for HTTP responses, XML for legacy partners, and protobuf for internal RPC. You'll add formats over time and add types over time.

Two designs are on the table:

  1. A Serializer<T> per (type, format) pair: OrderJsonSerializer, OrderXmlSerializer, UserJsonSerializer, etc.
  2. A visitor-based codec: each domain type implements accept(Visitor); each format is a Visitor.

The question isn't "which is better" — it's "when does each win?" Walk through the trade-offs.

Clarify
  • How many types do we expect (10? 100? 1000)?
  • How many formats (2? 5? 10)?
  • Is performance critical (RPC hot path) or is correctness/expressiveness more important?
  • Are domain objects allowed to depend on the serialization framework, or must they stay pure?
Approach
▸ Design thinking

This problem is explicitly about trade-offs, not about which design is "correct." The interviewer wants to hear you walk through both options and articulate when each wins.

Approach 1: Serializer<T> per (type, format) pair.

  • Pros: each serializer is independent, simple, fully customizable. Adding edge-case formatting for one type is trivial. No shared abstraction means no constraints on how each serializer works internally.
  • Cons: combinatorial explosion. With 50 types and 4 formats, you're maintaining 200 serializers. Adding a fifth format means writing 50 new serializers. Adding a 51st type means writing 4. The marginal cost grows linearly with breadth.
  • When it wins: few types, few formats, complex per-type formatting requirements that don't fit a generic visitor.

Approach 2: visitor-based codec.

  • Pros: cost is O(types + formats), not O(types × formats). Adding a new type means writing its accept; adding a new format means writing one new Visitor. Linear in the worst case; constant for "make existing types work in a new format."
  • Cons: domain types must couple to the visitor framework (they implement accept(Visitor)). This is OK for app-level types; it's a problem for library types you don't control. Also: the visitor abstraction has to anticipate all the field shapes you'll need (primitives, lists, nested objects). Real codecs handle this with overloaded field() methods or generic visitors.
  • When it wins: many types, many formats, types under your control. This is what Jackson, serde, protobuf-codegen all use.

The third option, worth mentioning: annotation- or reflection-based. The domain type is plain (no accept); the codec uses reflection or compile-time codegen to walk the fields. Same cost shape as visitor (O(types + formats)) without the coupling. Trade-off: reflection has runtime cost; codegen has build-system cost. Jackson does both; protobuf does codegen.

The senior answer: which one you pick depends on the type/format ratio, your control over the domain types, and your tolerance for framework coupling. There's no universal winner. The question to ask the interviewer is "how many types, how many formats, are the types under our control?" — and pick from there.

▸ Discussion / Follow-ups (with answers)
  • When does the visitor approach fail? When you can't predict the field shapes upfront. A Visitor with overloads for (String, int, List) handles 90% of cases; nested objects, polymorphic fields, custom types break the pattern unless you make the visitor generic. Generic visitors get unwieldy fast. Reflection-based codecs sidestep this — they look at the actual type at runtime.
  • What about backwards compatibility — old clients reading new data? This is where format-specific concerns leak in. JSON: ignore unknown fields by default (Jackson does). XML: depends on schema. Protobuf: explicit field numbers, ignore unknown numbers, never reuse field numbers. Encoding-level compatibility is a property of the format, not the codec, but the codec needs to expose the right knobs (e.g., Jackson's FAIL_ON_UNKNOWN_PROPERTIES).
  • Should domain types know about serialization at all? Strict separation says no — domain types are pure; codec is separate. In practice, this is purist and leads to either reflection (which couples by reflection rather than by interface) or external mappers (which duplicate type structure). Pragmatic answer: light annotations are OK (@JsonProperty), heavy framework integration is bad. Don't extend a framework class; do annotate fields if needed.
  • What about codegen vs reflection? Protobuf, Cap'n Proto, FlatBuffers use codegen — schemas are compiled to classes. Pros: fast, type-safe, schema is authoritative. Cons: build-system complexity. JSON libraries (Jackson, nlohmann::json) use reflection or templates. Pros: works with arbitrary types. Cons: slower, harder to enforce schema. Performance-critical RPC: codegen. Application-level JSON: reflection-based libraries.
  • How do you handle versioning of the wire format? Version field in the payload, or version in the metadata (HTTP header). The codec selects the right deserialization path based on version. Forward compatibility (old code reading new data) is usually easier than backward (new code reading old data); design with that in mind.
  • What if you need round-trip equivalence (deserialize then serialize gives identical bytes)? Hard. Most codecs don't preserve field order, whitespace, or representation choices. If you need this (canonical signing, content addressing), use a format with a canonical serialization (canonical JSON, JCS) or sign the payload before serialization rather than the serialized form.
  • What's a real pitfall with Approach 1 (per-pair Serializer)? Inconsistency between formats. OrderJsonSerializer and OrderXmlSerializer can drift — one includes a field the other doesn't. Tests catch some of this; not all. Approach 2 (visitor) and Approach 3 (reflection) keep one source of truth (the type itself).

Audit log with observers

~40 min Composition Medium
Prompt

Build an audit log for a multi-tenant SaaS. Every notable action — logins, permission changes, data access — gets recorded. Multiple downstream consumers need every event: the SIEM ships them off-prem, an alerting service watches for suspicious patterns, an analytics pipeline aggregates them. The audit log itself shouldn't know what its consumers do.

Design the interfaces. The interviewer will ask: what happens when a slow consumer blocks others? When one throws? When you need to add a new consumer at runtime?

Clarify
  • Synchronous or asynchronous notification?
  • What ordering guarantees do consumers need — strict, per-actor, none?
  • Is at-least-once OK, or is exactly-once required?
  • What's the retention policy for the events themselves?
Approach
▸ Design thinking

The shape: publisher–subscriber (observer) pattern with a clean async-or-sync choice.

The audit log accepts events and notifies subscribers. Subscribers register interest and process events. The audit log doesn't know what subscribers do; subscribers don't know about each other. This is the textbook observer.

The interesting design decisions are around what happens when things go wrong, because real systems have slow consumers, buggy consumers, and consumers that come and go.

Decision 1 — sync vs async dispatch.

  • Sync: record() calls each subscriber inline. Simple. But: a slow subscriber blocks the writer. A SIEM that takes 50ms slows every event by 50ms.
  • Async: record() appends to an internal queue; a worker thread drains the queue and notifies subscribers. Writers don't block. Trade-off: events can be lost on crash if the queue is in-memory.

For an in-process audit log, sync is usually fine and simpler. For a high-throughput service, async with a bounded queue (drop or block on overflow — explicit choice) is the right answer. Mention both.

Decision 2 — error isolation. One bad subscriber must not break the others. Wrap each notification in try/catch; log the error; continue with the next subscriber. The example does this.

Decision 3 — concurrent subscriber list. Subscribers can register at runtime. The list is read every record(), modified rarely. Two correct answers:

  • CopyOnWriteArrayList (Java) — readers see a stable snapshot without locking; writes are O(N) but rare.
  • Lock around both reads and writes, but copy the list before iterating to release the lock fast.

The first is the canonical answer for this pattern; the second works in C++ where there's no built-in CoW list.

Stateful subscribers are the interesting part. The example's SuspiciousAccessAlerter tracks per-actor failure counts; it fires on threshold. This is observer's real power — subscribers can be arbitrary stateful processors. Each one is a separate concern; each one is independently testable.

The follow-up that distinguishes: "how do new consumers get historical events?" Sync observer notifies only on new events. If a consumer comes online after some events have flowed, those events are gone. Two answers: (a) accept it (consumers must be running to receive); (b) keep an event log and replay on subscription. Real audit systems use (b) with append-only event logs (Kafka topics, transactional outbox + DB). Worth mentioning even if you don't implement.

Solution
// Audit log with observers.
import java.time.Instant;
import java.util.*;
import java.util.concurrent.CopyOnWriteArrayList;

public final class Audit {

    public record AuditEvent(String actor, String action, String resource, Instant ts) {}

    // Subscribers receive every event. Log doesn't care what they do.
    public interface AuditSubscriber {
        void onEvent(AuditEvent e);
    }

    public static final class AuditLog {
        // CopyOnWriteArrayList: writes rare (startup), reads (dispatch) hot.
        // Reads see a stable snapshot without locking.
        private final List<AuditSubscriber> subscribers = new CopyOnWriteArrayList<>();
        private final List<AuditEvent> events =
            Collections.synchronizedList(new ArrayList<>());

        public void subscribe(AuditSubscriber s) { subscribers.add(s); }

        public void record(AuditEvent e) {
            events.add(e);
            // Notify outside any lock — bad subscriber must not break others
            for (AuditSubscriber s : subscribers) {
                try { s.onEvent(e); }
                catch (RuntimeException ignored) {}
            }
        }
    }

    public static final class StdoutSink implements AuditSubscriber {
        public void onEvent(AuditEvent e) {
            System.out.printf("[audit] %s %s %s%n", e.actor(), e.action(), e.resource());
        }
    }

    // Stateful subscriber — tracks per-actor failure counts.
    public static final class SuspiciousAccessAlerter implements AuditSubscriber {
        private final Map<String, Integer> failedReads = new HashMap<>();
        public void onEvent(AuditEvent e) {
            if ("read.denied".equals(e.action())) {
                int n = failedReads.merge(e.actor(), 1, Integer::sum);
                if (n >= 3)
                    System.out.printf("  [alert] %s has %d denied reads%n", e.actor(), n);
            }
        }
    }
}

▸ Discussion / Follow-ups (with answers)
  • How do you guarantee delivery to subscribers? The in-process pattern here is at-most-once on crash (events in flight are lost). Production audit logs typically write to durable storage (DB, append-only file, Kafka topic) before notifying subscribers. Subscribers read from the durable store; they can replay from a position on restart. This makes delivery at-least-once with explicit replay semantics.
  • What about ordering? Sync dispatch preserves the order of record calls, but only if subscribers are sync too. Async dispatch with a single worker thread preserves order; multiple workers may reorder. If subscribers depend on order, document it and don't break it. For per-actor ordering (events for actor X are ordered, no ordering across actors), partition the queue by actor and have one worker per partition.
  • How do you handle backpressure on slow subscribers? If sync, slow subscribers slow everyone. If async with a bounded queue, three policies: (a) Drop oldest — newer events overwrite older; you lose data. (b) Block writers — backpressure flows back; writers slow down. (c) Spill to disk — overflow goes to a file; processed eventually. For audit specifically, dropping is wrong — audit data is the thing you don't lose. Block or spill.
  • How do you add a new subscriber that needs historical events? Sync observer can't help; events that already happened are gone. Two options. (a) Tell the new subscriber "you only see events from now"; they accept it. (b) Persist events; the new subscriber reads from the persistence on startup, then transitions to live. Real audit systems do (b) — the durable log is the source of truth, subscribers are projections.
  • Should subscribers run in the same process as the publisher? Depends. In-process is faster, simpler, and good for fan-out within a service. Cross-process (Kafka, RabbitMQ) is right for fan-out across services or different teams. The publisher's API can stay the same — a subscriber in-process is a callback; a subscriber out-of-process is a message bus producer that wraps the message and ships it.
  • How do you test stateful subscribers like the alerter? Unit-test them in isolation: feed events directly, assert behavior. Don't test through the audit log — that couples your alerter test to the log's wiring. The cleaner pattern: subscribers are pure functions of their inputs; testable without any of the publisher's machinery.
  • What about exactly-once semantics? In a single process with sync dispatch and no crashes, you have it. In a distributed system with retries, you don't — at-least-once + idempotent subscribers is the realistic answer. Make subscribers idempotent (using event IDs as deduplication keys) rather than chasing exactly-once delivery, which is much harder.

A Money type, the right way

~40 min Domain Medium
Prompt

You're working in a billing service that handles money in 14 currencies. The codebase uses double for amounts and string concatenation for currency. There are bugs related to floating-point rounding, accidental USD+EUR addition that silently produced nonsense, and an incident last month where a refund was off by one penny per transaction across several thousand transactions.

Design a Money value type that prevents these bugs by construction. Walk through: what operations does it support, what does it forbid, what's the unit, what about rounding, what about splitting a bill three ways without losing pennies.

Clarify
  • What's the smallest unit (cents? hundredths of cents for FX trading?)
  • What rounding mode is appropriate — half-up, half-even (banker's), truncate?
  • Is multiplication by non-integer scalars (tax rates, percentages) needed?
  • Should equality compare currencies, or just amounts?
Approach
▸ Design thinking

This problem is about value types — how to design a type whose instances represent values, not entities. Value types are equal by content; they're immutable; they encode their own rules.

What Money owes its callers:

  1. Exact arithmetic. No floating-point. Store as integer (cents, or microcents for FX). All arithmetic is on integers. 0.1 + 0.2 = 0.3, exactly, by construction.
  2. Type-safe currency. USD + EUR must be a compile-time or runtime error, not a silent nonsense result. Every Money carries its currency; every operation checks that the currencies match.
  3. Immutability. a.plus(b) returns a new Money; a and b are unchanged. This is what makes Money safe to share across threads and easy to reason about.
  4. Explicit rounding. When you multiply by a non-integer (a tax rate), what's the rounding rule? Half-up? Half-even (banker's)? Truncate? Picking one and documenting it is part of the type's contract.
  5. Conservation under split. Splitting $10 three ways must give back $10 in pennies, not $9.99. The "remainder distribution" algorithm — each gets the floor share, leftover pennies go to the first parties — is the standard answer. Some systems randomize which parties get the extra penny; pick a policy and stick with it.

What Money should refuse to do:

  • Be constructed from a string without explicit parsing. Money.parse("$10.00") is a separate factory; Money.of(amount, currency) is the primary constructor.
  • Be implicitly compared to numbers or other currencies. Equality requires same currency AND same amount.
  • Be multiplied by another Money. Currency × currency isn't meaningful. Multiplication is by a scalar.

The callers' bug fixes:

  • "Refund off by one penny": was almost certainly floating-point rounding. Money's integer arithmetic eliminates it.
  • "USD + EUR silently produced nonsense": Money's requireSameCurrency check prevents this; the program now throws instead of returning bad data.
  • "Off by N×1¢ across thousands of transactions": classic accumulated rounding. With integer cents, errors accumulate to zero.

Production references: Java's Joda Money library is the canonical reference. java.math.BigDecimal is what most homegrown solutions use under the hood (with a currency wrapper). C++ has no standard; boost::multiprecision::cpp_dec_float + a wrapper class is the usual approach. Mention these — it shows you've seen the problem before.

Solution
// A correct Money value type.
import java.util.ArrayList;
import java.util.List;

public final class Money {

    public static class CurrencyMismatch extends RuntimeException {
        public CurrencyMismatch(String m) { super(m); }
    }

    private final long cents;             // smallest unit
    private final String currency;

    private Money(long cents, String currency) {
        this.cents = cents; this.currency = currency;
    }

    public static Money of(double major, String currency) {
        // Half-away-from-zero. Defining rounding explicitly is part of
        // the type's contract.
        long c = Math.round(major * 100);
        return new Money(c, currency);
    }
    public static Money cents(long cents, String currency) {
        return new Money(cents, currency);
    }

    public long cents() { return cents; }
    public String currency() { return currency; }

    public Money plus(Money o) {
        requireSameCurrency(o);
        return new Money(cents + o.cents, currency);
    }
    public Money minus(Money o) {
        requireSameCurrency(o);
        return new Money(cents - o.cents, currency);
    }
    public Money times(int n) { return new Money(cents * n, currency); }

    @Override public boolean equals(Object o) {
        return o instanceof Money m && m.cents == cents && m.currency.equals(currency);
    }
    @Override public int hashCode() {
        return Long.hashCode(cents) * 31 + currency.hashCode();
    }

    // Allocate without losing pennies. Floor share + leftover pennies
    // distributed one-each. $10 split 3 ways = 3.34 + 3.33 + 3.33 = 10.00.
    public List<Money> split(int parties) {
        if (parties <= 0) throw new IllegalArgumentException("parties");
        long base = cents / parties;
        long remainder = cents % parties;
        List<Money> out = new ArrayList<>(parties);
        for (int i = 0; i < parties; i++) {
            out.add(new Money(base + (i < remainder ? 1 : 0), currency));
        }
        return out;
    }

    @Override public String toString() {
        return String.format("%d.%02d %s", cents / 100, Math.abs(cents % 100), currency);
    }

    private void requireSameCurrency(Money o) {
        if (!currency.equals(o.currency))
            throw new CurrencyMismatch(
                "cannot operate on " + currency + " and " + o.currency);
    }
}

class CurrencyMismatch : public std::runtime_error {

class Money {

public:

        : cents_(cents), currency_(std::move(currency)) {}

    static Money of(double major, const std::string& currency) {
        // round half-away-from-zero
        double scaled = major * 100;
        int64_t c = static_cast<int64_t>(scaled + (scaled >= 0 ? 0.5 : -0.5));
        return {c, currency};
    }

    int64_t cents() const { return cents_; }
    const std::string& currency() const { return currency_; }

        require_same_currency(o);
        return {cents_ + o.cents_, currency_};
    }
    Money operator-(const Money& o) const {
        require_same_currency(o);
        return {cents_ - o.cents_, currency_};
    }
    Money operator*(int multiplier) const {
        return {cents_ * multiplier, currency_};
    }
    bool operator==(const Money& o) const {
        return cents_ == o.cents_ && currency_ == o.currency_;
    }
    bool operator<(const Money& o) const {
        require_same_currency(o);
        return cents_ < o.cents_;
    }

    // Allocate a total across N parties without losing pennies.
    // Standard "remainder distribution" algorithm: each party gets the
    // floor share; the leftover pennies are distributed one each to the
    // first parties.
    std::vector<Money> split(int parties) const {
        if (parties <= 0) throw std::invalid_argument("parties");
        int64_t base = cents_ / parties;
        int64_t remainder = cents_ % parties;
        std::vector<Money> out;
        for (int i = 0; i < parties; ++i) {
            out.emplace_back(base + (i < remainder ? 1 : 0), currency_);
        }
        return out;
    }

    std::string toString() const {
        return std::to_string(cents_ / 100) + "." +
               (cents_ % 100 < 10 ? "0" : "") + std::to_string(cents_ % 100) +
               " " + currency_;
    }

private:
    void require_same_currency(const Money& o) const {
        if (currency_ != o.currency_)
            throw CurrencyMismatch(
                "cannot operate on " + currency_ + " and " + o.currency_);
    }

▸ Discussion / Follow-ups (with answers)
  • Why integer cents over BigDecimal? Both work; they have different trade-offs. Integer cents: fast arithmetic, can't represent sub-cent (problem for FX trading), simple equality. BigDecimal: arbitrary precision, slower, equality semantics are tricky (0.10 ≠ 0.1 by default — use compareTo). For application-level money, integer cents are usually right; for FX trading or scientific finance, BigDecimal or a custom decimal type. C++ doesn't have BigDecimal in std; libraries like boost::multiprecision::cpp_dec_float fill the gap.
  • What rounding mode is "right"? Depends on the use case. Half-up is intuitive but biased upward. Half-even (banker's) minimizes statistical bias when rounding many values; most accounting standards use it. Truncate is sometimes used for "we always round down to be safe." Pick based on jurisdiction and document it. Whatever you pick, encode it in the type's API; don't make callers think.
  • Why does multiplication take an int, not a Money? Because Money × Money isn't dimensionally meaningful. 5 USD × 3 USD would be USD², which doesn't exist. Multiplication is by a scalar (a count, a rate, a tax percentage). Rejecting Money × Money at the type level is the same instinct as rejecting USD + EUR.
  • How do you handle FX conversion? Make it explicit and external to Money. A CurrencyConverter service takes (Money, targetCurrency) and returns Money in the new currency. The FX rate is its own concern — versioned, time-stamped, sourced from a feed. Money itself doesn't know how to convert; if it did, every caller would need to inject an FX service into Money construction, which is wrong.
  • What about negative amounts (debits, refunds)? Allow them. Refunds are negative charges. Banking apps use signed amounts everywhere. Some systems use a Money type that's always non-negative and a separate Direction (DEBIT/CREDIT) field. Either works; signed is simpler unless the domain insists on the explicit direction.
  • How do you serialize Money? As {"amount": 12345, "currency": "USD"} with the integer in cents. Avoid {"amount": 123.45} — JSON's number type is double-precision float. The smallest-unit-as-integer convention is what every bank API uses (Stripe, PayPal, Plaid all serialize money this way).
  • What's the existing-code migration path? Strangler fig. Introduce Money alongside double. New code uses Money; old code unchanged. Migrate callers one at a time, starting with the most critical paths (payment, refund). Run reconciliation between the old and new computations during the migration; alerts on disagreement. Once all callers are migrated, delete the doubles. Don't try to swap everything at once; the change set is too big to review safely.

Refactor a UserService that does too much

~50 min Refactor Hard
Prompt

You inherit a UserService class. It has methods register, login, changePassword, resetPassword, updateEmail, deactivate, listUsers, countActiveUsers, sendNewsletter, and 14 other methods. It depends on the database, the email service, the SMS service, the audit log, the analytics service, the cache, and the feature flag system. Tests need a sea of mocks. Two engineers blocked each other on it last week.

Walk through the redesign. The interviewer is testing whether you can articulate which kinds of cohesion matter and what shape the replacement should have.

Clarify
  • Is this brownfield (must preserve API) or greenfield-style refactor?
  • Is CQRS-style separation of reads and writes acceptable, or single-class style?
  • Do callers depend on UserService directly, or via an interface that can stay?
  • Is async/event-driven on the table?
Approach
▸ Design thinking

This is the same problem as #5 with extra layers, but the diagnosis is different. OrderProcessor failed because it mixed too many infrastructure concerns. UserService fails because it mixes too many use cases. Worth being able to articulate the difference.

The diagnosis:

  • UserService has 24 methods. Some are commands (register, login, change password); some are queries (count, list); some are unrelated (sendNewsletter — what is that doing here?).
  • Each method touches a different subset of the dependencies. Login uses repo, hasher, tokens, audit. Register uses repo, hasher, email, audit. Newsletter uses email and... a query against the repo. The shared dependencies between these is much smaller than the union.
  • Tests need a sea of mocks because every test method has to set up all dependencies (constructor takes them all), even though it only uses a few.
  • Two engineers conflict because they're editing different methods that happen to share the same file.

The frame for the redesign: cohesion by use case, not by entity. The instinct is to organize code around nouns ("everything about Users goes in UserService"). The better instinct is to organize around verbs ("everything needed to register a user goes in RegisterUser"). This is sometimes called the Use Case pattern (Clean Architecture), or Command Handler pattern, or Application Service pattern. Same idea: one class, one job.

The new structure:

  • RegisterUser — depends on UserRepository, PasswordHasher, EmailSender, AuditLog. ~15 lines.
  • LoginUser — depends on UserRepository, PasswordHasher, TokenIssuer, AuditLog. ~15 lines.
  • ChangePassword — depends on UserRepository, PasswordHasher, AuditLog. ~10 lines.
  • UserQueries — read-only queries: count, list. Depends on a read model or repository.
  • NewsletterSender — has nothing to do with users; it gets its own home.

What this buys you:

  • Each test sets up only the dependencies it needs. No mocks for unrelated things.
  • Two engineers editing register and login don't conflict — different files.
  • Adding a new use case (e.g., "verify email") is a new class, not a new method on a god-class.
  • Reading the codebase, "what does the system do?" is answered by listing files in the use case directory.

The trade-off worth being honest about: more files. A team that's used to "everything in one Service" finds this disorienting. The 24-method class becomes 24 files (or use cases bundled by group). The senior judgment call is: if the team is small and the system is small, the file count is overhead. If either grows, the use-case-per-class shape pays back. Most production systems benefit from it.

CQRS as the natural extension: separate the command side (RegisterUser, LoginUser, ChangePassword) from the query side (UserQueries). They have different concerns — commands write and validate, queries read and join. Commands can use a write-optimized repository; queries can use a denormalized read model. Mention this as the next step beyond use-case extraction; don't necessarily implement it.

▸ Discussion / Follow-ups (with answers)
  • Cohesion by entity vs cohesion by use case — why does this matter? Entity-cohesion ("everything about Users in UserService") seems organized but groups together code that doesn't actually share dependencies or change reasons. Login and sendNewsletter both touch User but are otherwise unrelated — they're co-located only because of the entity. Use-case-cohesion ("everything to do this one job in one class") groups code that actually changes together. This is what people mean by the Single Responsibility Principle — single reason to change, not single noun.
  • Aren't 24 use case classes harder to navigate than one service? Different kind of harder. A 24-method service is one big file you scroll through, mentally tracking which method does what. 24 classes are a directory you list. With good naming (RegisterUser, LoginUser, ChangePassword), the directory listing tells you what the system does. Most teams find this easier after a brief adjustment period.
  • What about shared code between use cases? Extract it. Common validation, password rules, audit conventions go into helpers or shared services. The use case classes become thin orchestrators; the shared logic is in dedicated places. Don't duplicate; don't shoehorn shared logic into a base class either — composition over inheritance applies here too.
  • How do queries fit? Reads are different from writes. List, count, search are queries; they don't have side effects, don't need audit, don't need transactions. CQRS is the natural fit: write side is use cases (RegisterUser, LoginUser); read side is queries (UserQueries.list, UserQueries.count). The two sides may even use different data models — write side normalized, read side denormalized for query performance.
  • What about transactions across use cases? One use case = one transaction. If "register user" needs to atomically (a) create the user and (b) send a welcome email, that's two concerns: the DB write must be transactional (insert user + audit log in one tx); the email send must be retried independently (transactional outbox pattern, see problem 8). Use cases handle the first; infrastructure handles the second.
  • How do you avoid the use case classes becoming anemic delegators? They will be small — that's the point. The "logic" of "register a user" is genuinely small: validate, hash password, save, send email, audit. Five lines. If the use case is anemic, that's a feature, not a bug — it means the work is happening in well-named, separately-testable collaborators. The thing to avoid is use cases that just call one thing — that's a layer with no value. The orchestration of multiple collaborators is the value.
  • Brownfield migration? Strangler fig again. Keep UserService as a facade that delegates to the new use case classes. New code calls the use cases directly; old callers keep using UserService. Over time, migrate callers; eventually delete UserService. Don't try to delete the god-class on day one — the change set is enormous and reviewers will rubber-stamp it.
